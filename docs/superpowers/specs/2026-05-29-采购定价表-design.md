## 1. 背景与目标

现有采购流程（`app/routes/purchase.py` + `app/services/purchase.py` + `static/js/purchase.js`）已能：
上传供应商 Excel → 解析条码/进价/数量 → 识别 NEW 新品与匹配老品 → 投票出供应商 → 生成系统导入模板。

**缺口**：没有任何「进价变动」对比逻辑；现有导出是给系统用的导入文件，而不是给老板做定价决策的报价表。

**本功能目标**：在采购上传流程之上，新增「生成老板定价表」——
1. 算出该供应商老产品的典型利润率（中位数），供用户确认/调整为**目标利润率**；
2. 为本次**新品**按 `批发价 = 进价 ÷ (1 − 目标利润率)` 算出建议批发价；
3. 为本次**进价有调整的老品**列出 旧进价→新进价、现售价、现利润率、建议新售价；
4. 生成**单个 xlsx**（带活公式、留图片列、含热度参考），下载发给老板定价。

> **Goal Translation**
> - action: 实现采购定价表生成 / verify: `pytest tests/test_pricing_sheet.py` 全绿（中位数、调价检测、批发价公式、xlsx 结构与公式回读）
> - action: 接入采购页前端 / verify: 本地 `python server.py` 上传样例 → 点「生成定价表」→ 确认利润率 → 下载 xlsx，openpyxl 回读断言两段与公式

## 3. 端到端流程

```
上传供应商 Excel（现有）
   └─ 解析 条码/进价/数量，识别 NEW + 匹配老品，投票供应商（现有）
   └─ 在 Section 03 填新品中文名（现有）
        ↓ 新增按钮「生成老板定价表」
POST /purchase/pricing/preview  { rows, new_entries, supplier_id }
   ├─ target_margin = median( 该供应商在售老品 margin_pct )
   ├─ changed_old = 匹配老品中 upload_price ≠ last_purchase_unit_price 的
   └─ → { target_margin | null, n_samples, n_new, n_changed, skipped_no_baseline }
        ↓ 前端确认框：利润率 [30.0]%（来自 N 个老品中位数）｜新品 X｜调价 Y
          target_margin=null → 留空 + 提示「无足够历史利润数据，请手填」
        ↓ 用户确认/改利润率
POST /purchase/pricing/export  { rows, new_entries, supplier_id, target_margin }
   └─ 后端重算 items + openpyxl 生成 → send_file 下载
      文件名：定价表_{供应商}_{YYYYMMDD}.xlsx
```

> export 不信任前端传来的计算值：用与 preview 相同的输入 + 用户确认的 `target_margin` 在**服务端重算**全部条目，保持单一真相源。

### 4.2 单元格与公式（设 1-indexed；目标利润率存 `B2`）

| 列 | 表头 | 新品段 | 调价老品段 | 公式/格式 |
|----|------|--------|-----------|-----------|
| A | 图片 | 空 | 空 | 列宽≈18，行高≈90（手动拖图） |
| B | 条码 | upload barcode | upload barcode | 文本 `@` |
| C | 中文品名 | new_entries 的品名 | `stockpile.product_name_zh` | — |
| D | 数量 | upload qty | upload qty | `0` |
| E | 旧进价 | 空 | `last_purchase_unit_price` | `#,##0.00 "€"` |
| F | 新进价 | upload price | upload price | `#,##0.00 "€"` |
| G | 现售价 | 空 | `stockpile.sale_price` | `#,##0.00 "€"` |
| H | 现利润率 | 空 | `=IF(AND(G{r},F{r}),(G{r}-F{r})/G{r},"")` | `0.00%` |
| I | 建议批发价 | `=F{r}/(1-$B$2)` | `=F{r}/(1-$B$2)` | `#,##0.00 "€"` |
| J | 建议利润率 | `=IF(I{r},(I{r}-F{r})/I{r},"")` | 同 | `0.00%`（构造上=目标） |
| K | 修改 | 空（老板填） | 空（老板填） | `#,##0.00 "€"` |
| L | 改后利润率 | `=IF(K{r},(K{r}-F{r})/K{r},"")` | 同 | `0.00%` |
| M | 热度 | 文本"新品" | `urgency_score`（None→空） | `0` |

- **H 现利润率用新进价 F**：体现成本变化后、若维持现价的利润挤压。
- **I 引用 `$B$2`**：改 B2 → 全表建议批发价重算。
- 表头：加粗 + 灰底 `PatternFill("solid", fgColor="E8E8E8")` + 居中；冻结到首段表头行。
- 段分隔行（"◆ 新产品" / "◆ 调价老产品"）：跨列合并、底色区分。

## 5. 后端结构（遵循项目分层规范）

### 5.1 新服务 `app/services/pricing_sheet.py`

- `compute_supplier_target_margin(supplier_id, session) -> {"median": float|None, "n_samples": int}`
  取该供应商在售老品的 `margin_pct`，求中位数；样本不足 → `median=None`。
  实现：从 `analytics.list_sku_summary`（60s 缓存）的 `barcode→item` 映射中，取属于该供应商的条码（`stockpile.supplier_id==sid`，回退到最近采购供应商）的 `margin_pct`（非 None 者）。

- `detect_changed_old_items(rows, system_set, session) -> list[dict]`
  上传里在系统内的条码，`upload_price != last_purchase_unit_price` → 列入；`last_purchase_unit_price is None`（无基准）→ 跳过并计数。

- `build_pricing_items(rows, new_entries, supplier_id, session) -> {"new": [...], "changed": [...], "skipped_no_baseline": int}`
  新品段（名取 new_entries）+ 调价老品段（补 `product_name_zh`、`sale_price`、`urgency_score`）。

- `build_pricing_xlsx(items, target_margin, supplier_name, date_str) -> bytes`
  openpyxl 全内存（`io.BytesIO`），按 §4 写两段 + 活公式 + 样式。

### 5.2 新 repo helper `app/repositories/stockpile_db.py`

- `query_supplier_products(supplier_id) -> list[Row]`：返回该供应商在售产品（barcode/model/name_zh/stock_price/sale_price/last_purchase_unit_price/origin）。探查确认此查询目前缺失，需新增。

### 5.3 新路由（加进 `app/routes/purchase.py` 现有蓝图）

- `POST /purchase/pricing/preview` → `{ ok, target_margin|null, n_samples, n_new, n_changed, skipped_no_baseline }`
- `POST /purchase/pricing/export` → `send_file` 回 xlsx（`application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`，`as_attachment=True`）。

### 5.4 复用
`lookup_dominant_supplier`（供应商）、`analytics`（margin/urgency）、`stockpile.last_purchase_unit_price`、现有 openpyxl 样式套路（scraper 输出/`build_output_excel`）。

## 6. 前端（`static/js/purchase.js` + 采购页模板）

- 解析结果区新增按钮「生成老板定价表」（在已有「导入/下载」附近）。
- 点击 → `POST /purchase/pricing/preview` → 弹确认框：
  - 可编辑利润率输入框（预填中位数；null → 空 + 提示「无足够历史利润数据，请手填」）
  - 一行摘要：`新品 X 个 · 调价老品 Y 个`（`skipped_no_baseline>0` 时补「N 个老品无历史进价，未列入」）
- 确认 → `POST /purchase/pricing/export` → 触发浏览器下载（blob）。
- 无新品且无调价老品 → 不弹框，提示「本次无新品也无调价，无需定价表」。

## 7. 错误处理 & 边界

- 无新品且无调价老品 → 不生成，前端提示。
- 利润中位数算不出 → 确认框留空 + 提示，用户手填后才能导出。
- 供应商未识别（全新品）→ 标题用 Section 03 用户填的供应商名；目标利润率走手填。
- 老品 `last_purchase_unit_price` 为空 → 不计入调价段；计数并在摘要提示。
- `urgency_score` 为 None（新近/停售）→ 热度列留空。
- **货币假设**：上传进价按 **EUR**（与现有采购解析一致）。若某 CN 供应商上传为人民币，属边界，**暂不处理，标注待确认**。
- target_margin 取值校验：`0 ≤ target_margin < 1`（否则 1/(1−x) 异常）；前端+后端双校验。

## 8. 测试（`tests/test_pricing_sheet.py`，纯函数优先）

- 中位数：正常 / 含异常值（中位数抗偏）/ 样本不足→None。
- 调价检测：相等不列 / 不等列 / 基准为空跳过且计数。
- 批发价数值：`进价/(1−利润率)` 正确（多组）。
- xlsx 回读（openpyxl 重新打开生成的 bytes）：
  - 两段表头与分隔行存在；
  - I 列公式串含 `$B$2`；J/H/L 公式正确；
  - 数字格式（€/百分比）正确；
  - 图片列 A 为空、行高加高；
  - 新品段 E/G/H 空、M="新品"。
- 边界：target_margin ≥ 1 被拒。

## 10. 非目标（YAGNI）

- 不做浏览器内实时预览表（已选后端确认式）。
- 不自动嵌入产品图片（用户手动拖）。
- 不做多供应商混合定价（一单一供应商）。
- 不持久化定价表历史（一次性下载）。
- 不处理上传文件的人民币货币换算（边界，待确认）。
