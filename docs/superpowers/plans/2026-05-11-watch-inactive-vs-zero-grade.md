> **不是 plan，是 watchlist**：要不要做、做哪一种，等真出问题再决定。

## 问题本质

主档 `stockpile` 表里有两套"停用"语义，互不耦合：

| 字段 | 谁写 | 含义 |
|---|---|---|
| `is_active=0` | **系统自动**（`stockpile_db.py:371 _deactivate_missing_records`）| 老 ERP 同步时这次 product.csv 里没出现这个货号 → 自动打 0 |
| `manual_grade=0` | **用户手工**（销售分析页）| "这个货我不再补货 / 业务上停用" |

两个字段独立 —— **系统下架不会清掉用户评级**，反之亦然。

## 那 15361 个 "is_active=0 + manual_grade=1-10" 是什么货？

> 你曾给等级 1-10（说明用过/熟），后来老 ERP 把它从 product.csv 里移除了。

典型场景：
- 老 ERP 把 `web_status` 改成 N（下架但不一定店里没货）
- 季节性货临时停售
- 老款 EOL 但仓里还有库存准备清

**业务上有歧义**：
- 视角 A："系统已下架 → 进货时别下单了"
- 视角 B："只是老 ERP 状态没维护好 → 还能卖"

代码层面老注释自己承认：`"inactive ≠ 店里没货"`（`stockpile_db.py:496`）。

## 触发"该处理了"的信号

哪天发生以下任一情况，就回来看这份 plan：

1. **一份采购 Excel 拖进来出现一堆这种货**（is_active=0 + grade=1-10 的多个 row 显示绿色 MATCH，但你心里其实觉得"这些是不是不该再进货"）
2. **下单了一批货才发现里面有不少老 ERP 已下架的**
3. **想做"补货建议"功能时**，要决定这一档算"在售"还是"已下架"
4. **数据质量页**想加"系统下架但有等级"的检测

## 备选方案（到时候挑一个）

**方案 A · 加第二档 pill**
新增 `DOWN`（灰）状态：`is_active=0 且 manual_grade != 0`。OFF 红 + DOWN 灰共存。
- 优：信息全，用户能区分系统下架 vs 业务停用
- 缺：UI pill 增到 4 档，复杂度上升

**方案 B · 把"系统下架"等价为"停用"**
OFF 改为 `manual_grade=0 OR is_active=0`。
- 优：保守稳妥，所有"可能不该再进"的都标
- 缺：可能误报很多还能卖的货

**方案 C · 数据清洁先**
跑一次脚本，把 `is_active=0 + manual_grade=1-10` 中确认 EOL 的批量设 manual_grade=0，让两个字段对齐后再决定。
- 优：解决根因
- 缺：要先建立"哪些算 EOL"的判断标准（销量 / 库存 / 时间窗口）

**方案 D · 维持现状**
OFF 只看 grade=0，is_active=0 的不特殊处理。
- 优：语义最干净（系统状态 ≠ 业务决策）
- 缺：用户可能不知道某些货已经被老 ERP 下架
