## 阶段 1.5：多库位 schema 改造（feature/multi-location）

**前置依赖**：阶段 1 完成（必须）。**没 ORM + Alembic 之前不要碰这个**。

### 决策已定（2026-04-29 会话）

1. **业务真相**：用户确认两类多库位场景
   - Case 1：旧位 + 新位过渡（不确定老位是否清空，保留两者）
   - Case 2：溢出多位（如 A14-12-01 + A14-12-02 同款多个货架）
   schema 不区分这两种，都用"一货号 N 库位"模型

2. **schema 选型**：**(a) 一对多子表**
   - `stockpile_locations(stockpile_id FK→CASCADE, location, kind, position, created_at)`
   - UNIQUE(stockpile_id, location)
   - 主表 `stockpile.stockpile_location` **永久保留**作月度比对的字节级源
   - 子表是 raw 字符串的派生分析视图

3. **历史噪音处理**：**观察 ≠ 修改**。**不写清理脚本**
   - 老系统的 round-trip / 空格噪音可能是真实业务事件，删了就丢证据
   - 改成"数据质量分析页"列出异常给用户看，老系统改完下次 import 自动同步过来

4. **比对功能保护**：用户提出顾虑——修复期内 cosmetic 不一致会淹没 substantive 真错误
   - 解决：**双轴比对**。compare 输出 cosmetic_mismatches + substantive_mismatches 两类
   - normalize 规则：每段独立 strip，**保留段顺序**（店面在前/仓库在后是契约）
   - substantive ≥ 3 触发显眼告警
   - 加 stockpile_snapshots 表，每次 import / compare 留计数快照供趋势分析

### 实施（拆 4 个 PR，全部完成）

## 决策日志（2026-04-28 会话）

按时间顺序记录这次会话产生的关键决策，未来想质疑某个选择时回看：

1. **路线分期**：长期路线拆成 4 个独立子系统（A 货号历史 / B 扫描历史 / C 进出货事件 / D 销售分析），每期独立 spec/plan/分支。**不做"先把所有数据建模做完再开始"那种瀑布**。
2. **第 1 期范围**：只做"货号历史搜索 + 库位变更时间线"，**只读**。明确不做：模糊搜索、就地编辑、扫描批次、销售事件、销售分析。
3. **聚合算法**：5 秒窗口聚合（基于实际数据时间间隔分布：0-5s 占 56%、6-60s 0%、>1min 44%）。
4. **轮回噪音**：发现 927 条 stockpile_changes 里 65%（602 条）是 import 流程的来回切噪音，根因在 `_sync_export_dataframe` 没去重。**本期不修**，留给阶段 1.5。
5. **多库位真相揭示**：`stockpile_location` schema 实际是"店面/仓库"两段编码（A/B/C 开头是店面，X/Z 开头是仓库），但 `parse_locations` 不允许同维度多个。用户确认业务上**真的存在**同时两个店面/两个仓库的货号，**当前 schema 表达不了**。需要阶段 1.5 重做。
6. **admin 清理**：`/admin` 控制台页 + `/stats` 月度扫描柱状图全删（用户从未实际使用 /admin，互传已通过 A 端 FAB 抽屉解决）。保留：`routes_collab` / `routes_transfer` / `routes_monthly_summary` / `dual_mode` 配置（A↔A 互传依赖）。
7. **技术栈重构方向**：用户提议 Tailwind + Alpine + ECharts + Clusterize + Flask + SQLAlchemy + WAL + Calamine。评估结论：**80% 价值集中在 SQLAlchemy + Alembic**（直接解决 schema 演进痛点）；其余各自有价值但不解决当前痛点，按需引入。**不一次性大爆炸重构**，分阶段。
8. **重构对现有功能的影响**：渐进迁移，逐模块 ORM 化，每个 PR 保持函数签名不变。**功能层面应该 0 影响**，但要提防 SQL 行为细微差异（is_active 1/True 比较、datetime 默认值、extra JSON 列）和性能回归（ORM 比 raw 慢 2-3x，热路径要 with_entities 优化）。
9. **第 1 期收尾**：选 (A) 当前分支用现有技术栈合掉，重构在新分支独立做。不在当前分支扩范围。
