# ADR-0001: 补货策略形式化 — (R,S) 周期盘点制

> **Status**: Accepted（2026-06-11 用户批准；lead time 先验定为 4 周）
> **Date**: 2026-06-11
> **作者**: Fable 5（判断层产出，实施由后续 plan 执行）
> **关联**: `docs/adr/replenishment-redlines.md`（红线清单）、`docs/superpowers/plans/2026-06-11-replenishment-correctness.md`（实施 plan）

## 背景与现状

系统已有一套**隐式补货策略**在生产运行（`app/services/analytics/restock_calc.py`），
本 ADR 的目的不是从零设计，而是把它形式化、指出缺口、决定补什么。

### 现行隐式策略（逆向自代码）

| 要素 | 现行实现 | 位置 |
|---|---|---|
| 盘点周期 R | 1 周（scraper 周更库存快照 + forecast_output 周刷） | `scraper/` + `refresh_forecast_output` |
| 目标水位 S | 8 周需求覆盖（`_RESTOCK_TARGET_WEEKS = 8`） | `restock_calc.py:242` |
| 订货量 | `ceil(周分位数 × 8) − 现库存`，clamp 非负，凑整到中包 | `restock_calc.py:254-300` |
| 服务水平旋钮 | p50 / p98 双列推荐（用户自选保守程度） | 同上 |
| 订货点 s | 无显式 s；urgency_score 启发式（销速/覆盖/久未购/毛利四项加权）排序代替 | `restock_calc.py:152-212` |
| 在途量 | **不扣减**（缺口） | — |
| lead time | **不进入计算**（缺口） | — |
| 需求输入 | `forecast_output` 表周分位数（EmpiricalQuantile，仅 retail/mixed） | `app/services/forecast.py` |

### 数据审计事实（2026-06-11，本地 PG 镜像线上）

决定设计空间的三个硬事实：

1. **lead time 历史分布当前不可得**。`purchase_orders` 表 2026-05-27 才开始记录
   （采购单 UI 上线），共 11 单、仅 1 个 `arrival_date` 样本。ERP 进货事件
   （`inventory_events`，event_type='purchase'）有 6 年历史（71,213 行，2020-02 起）
   但只有到货日、无下单日，反推不出 lead time。
2. **在途缺口是当前最大的实际偏差**。4 张 placed 状态采购单、473 行明细、
   **64,165 件在途、覆盖 459 个 SKU**（最大一张 53,677 件），全部不在补货计算内。
   字段齐备：`qty_ordered − qty_arrived`。
3. **库存快照历史仅 3 周**（7 个快照日，2026-05-20 起）。缺货周判定
   （`app/services/stockout.py`）只能覆盖近期，删失需求修正只能向前生效，
   随快照积累自动增强。

### 回测事实（run 44，2026-05-19）

- 生产模型 EmpiricalQuantile（base_demand 视图）：MAPE 0.814 / MASE 0.978 / **Cov@98 0.972**
- 点预测仅略优于 naive；**分布校准是这套系统真正可靠的部分** → 策略设计应消费分位数而非点估计
- 正态近似（μ+z·σ）的 Cov@98 在 0.886–0.926，被经验分位数全面压制 → 不走参数化安全库存公式

## 决策

### D1. 策略参数化：(R,S) 周期盘点制，R = 1 周

**选 (R,S)，不选 (s,Q) / (s,S) 连续盘点。**

理由：库存可见性是周批量的（scraper 周一抓快照），连续盘点制需要实时库存位置，
在这套系统里是虚构。R = 1 周不是设计选择，是数据节奏的事实，**写死并文档化**，
不做配置项（YAGNI：改 R 意味着改 scraper 节奏，那是另一个工程决策）。

### D2. 目标水位 S：保护期需求分位数，基于库存位置 IP

```
保护期 H = R + L                      （L = lead time，见 D4）
S      = Q_α(D_H)                     （D_H = H 周需求总和；α 见 D3）
订货量  = max(0, S − IP)              （凑整到中包，见 D6）
IP     = 现库存(qty_total, 负值取0) + 在途(Σ qty_ordered − qty_arrived, 非作废单)
```

**在途扣减是本 ADR 最高优先级的修复**（红线 RL-2）：64k 件在途不扣，
等于每周对同一缺口重复下单。

安全库存不单列：S 取的是经验分位数，`Q_α(D_H) − μ·H` 就是隐式安全库存。
不走 `SS = z·σ_D·√H + z·μ·σ_L` 教科书公式 —— 回测已证明正态近似在这批
间歇序列上尾部覆盖差（0.886 vs 0.972）。

### D3. 服务水平 α：保留 p50 / p98 双档展示，映射关系写明

- **p98 档**：周期服务水平 ≈ 98%（每个补货周期缺货概率 2%）。适合高毛利/高紧迫 SKU。
- **p50 档**：≈ 50% 周期服务水平，本质是"中位数需求刚好够"——适合低毛利、
  可接受缺货、资金占用敏感的 SKU。
- 用户按 urgency_score 和毛利自行选档（现行交互不变）。
- **未来扩展**（不在本期）：按 SKU 分层自动选 α（毛利分位 → α 映射表）。先不做：
  单操作员场景下人工选档的判断质量高于规则，自动化收益不明。

### D4. lead time L：配置先验 + 数据积累后自动切换经验分布

lead time 历史不可得（审计事实 1），因此：

1. **现在**：全局配置 `REPLENISH_LEAD_TIME_WEEKS`，**默认 4 周（用户 2026-06-11
   按实际供应链校准：中国采购 → 希腊到货约 4 周）**。
   配置进 `app/config.py` 或环境变量，不硬编码进算法。
2. **自动切换**：当 `purchase_orders` 中 `arrival_date` 非空的样本 ≥ 20 单时，
   切换为经验分布：L 取样本 **p90**（不取均值 —— lead time 右偏，均值低估风险），
   并在系统管理页展示当前用的是先验还是经验值、样本数。
3. **为什么 p90 不是 p50**：L 进入保护期 H，H 偏短的代价（缺货）远大于偏长的代价
   （多备 1-2 周库存）。在样本少时尤其要偏保守。

注意：L=4 周时 H = R + L = 5 周，**小于现行 8 周 target** —— 形式化后推荐量
在 horizon 维度也会下降（叠加 D5 的聚合修正），这是回到正确水位，不是回归。
target 周数退役，由 H 取代。

### D5. 跨期聚合：bootstrap 求和分位数，废除 周分位 × N

**现行错误**（红线 RL-1）：`周p98 × 8` 当 8 周需求上限、`周p98 × 13` 当季度 p98
（`restock_calc.py:274,289` 与 `compute_forecast_snapshot` 的 `quarter_p98`）。
分位数不可线性相加：i.i.d. 下 `Q_98(ΣD) ≈ 13μ + 2.05σ√13 ≪ 13 × Q_98(D)`，
周间波动互相抵消（√N 律）。间歇序列（多数周为 0）下高估最严重，可达 2-3 倍。

**修正**：对历史周序列做 bootstrap —— 有放回抽 H 个周值求和，重复 2000 次，
取和的经验分位数。约 20 行 numpy，单 SKU < 1ms，固定 rng seed 保证可复现。

**记录的已知限制**：bootstrap 假设周间独立（i.i.d.）。若需求有正自相关
（连续旺周），和的真实尾部比 bootstrap 厚。当前生产模型 EmpiricalQuantile
本身就是 i.i.d. 假设，此限制与模型一致，不额外引入误差。留给回测验证：
walk-forward 框架可直接评估 H 周和的覆盖率（coverage@p98 对 8 周窗口），
作为该修正上线的验收标准。

### D6. 反震荡触发阈值（s 的现代替身）

纯 (R,S) 每周只要 `S − IP > 0` 就触发订货，间歇 SKU 会出现每周下 1-2 件的
订单 churn。引入**下单触发条件**（不是教科书的 s，是工程阈值）：

```
触发下单 ⟺ S − IP ≥ max(1 个中包, 0.25 × S)    或    现库存 ≤ 0 且 S > 0
```

即：缺口不足一个中包或不足 S 的 1/4 时不提示补货（继续持有），除非已经断货。
0.25 系数为初值，可配置。这同时解决了"s 和 S 在间歇 SKU 上退化重合"的边界
（S 很小时缺口永远小于阈值 → 自然不触发，不会震荡）。

### D7. 删失需求：缺货周从训练序列剔除

缺货周销量 0 不是需求 0（删失）。0 入训练 → 预测被拉低 → 补货更少 → 更缺货，
死亡螺旋（红线 RL-3）。

**修正**：`_build_series`（`app/services/backtest.py:281`）构建序列时，
调 `stockout_weeks()` 把缺货周**剔除**（当缺失周处理，不是填 0 也不是插值）。

- 选剔除不选插值/仿真修正：EmpiricalQuantile 是 i.i.d. 模型，剔除等价于
  "对有货周的需求分布取样"，统计上干净；插值引入人造值污染分位数；
  仿真修正（论文二档加分项 A）是后续升级路径，不阻塞止血。
- 覆盖范围诚实声明：快照历史仅 3 周（审计事实 3），当前实际影响小，
  机制先立起来，随快照积累自动增强。`n_weeks_history` 语义变为"有效训练周数"，
  剔除数记入新字段 `stockout_weeks_excluded` 供置信分层消费。
- 对 walk-forward 回测的影响：剔除打破周连续性，对 i.i.d. 模型无影响；
  对 lag-52 类季节模型有影响 —— 但季节模型已退役（NaiveSeasonal52W 留作 baseline
  时用未剔除序列，对比口径在 run 元数据里记录 view 变体）。

### D8. 明确不做（YAGNI，单操作员 + 内网约束）

- **仓容上限约束**：无仓容数据，不建模。红线清单里保留"单次推荐量 > 历史最大
  单次进货量 × 3 则告警"的合理性闸代替。
- **多级库存 / 多仓**：单仓事实，不建模。
- **预算约束优化**（在预算内最优分配订货）：人工决策质量更高，不自动化。
- **per-SKU lead time**：每单一供应商（项目记忆），但供应商维度样本更稀，
  先全局 L，样本够了再分供应商。

## 后果

**正面**：
- 在途扣减消除重复下单（影响 459 个 SKU 的推荐值）
- bootstrap 聚合让 p98 档推荐量回到统计正确水位（预期普遍下降，间歇 SKU 降最多）
- 删失剔除阻断死亡螺旋的机制性成因
- 推荐量从"拍系数"变为"可被回测验收"（8 周和的 coverage 可量化）

**负面 / 风险**：
- 推荐量普遍下降初期会让用户感觉"系统变保守了"——需在 PR 描述和系统页说明原因
- bootstrap 引入随机性（已用固定 seed 消除不可复现性）
- `forecast_output` 需要加列（schema 迁移），restock 页消费逻辑同步改
- L 先验 4 周为用户经验值，PO 到货样本满 20 单后自动切经验 p90 自我修正

## 备选方案（已否决）

| 方案 | 否决理由 |
|---|---|
| (s,Q) / (s,S) 连续盘点 | 库存可见性周批量，连续盘点是虚构 |
| 正态近似安全库存公式 | 回测证明尾部覆盖差（0.886-0.926 vs 0.972） |
| 周分位 × N 维持现状 | 数学错误，高估 2-3 倍，见 D5 |
| 缺货周仿真修正需求 | 复杂度高一个量级，留作论文二档升级路径 |
| 自动 per-SKU 服务水平 | 单操作员人工选档质量更高，YAGNI |

## 附录：本次审计发现的隐式耦合（P5 素材，待并入 data-analytics-overview 刷新）

1. **周粒度 × 系数链**：forecast_output 存周分位 → restock 乘 8（target 周）→
   briefing/快照乘 13（季度口径）。三处隐式约定同一个周粒度，改任何一处其余静默错。
   本 ADR 落地后系数链消失（改为存 horizon 分位数列）。
2. **barcode ↔ product_model 桥接 hack**：`restock_calc._lookup_qty` 用
   "13 位条码取倒数第 2-6 位"匹配快照表的 product_model。两套主键体系的脆弱桥，
   `stockout.py` 则走 Stockpile 表查 model —— 同一映射两种实现。
3. **分类 → 需求视图耦合**：`base_demand_view` 的大单剔除策略按 sku_type 分流，
   改 categorizer 阈值会静默改变所有下游预测输入（回测结果随之失效，需重跑）。
4. **缺货判定依赖周一快照存在性**：无周一快照 → unknown → 不判缺货（保守）。
   scraper 漏抓一周 = 该周删失修正失效，无告警。
