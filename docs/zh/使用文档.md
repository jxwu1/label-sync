# 双端处理 使用文档

## 简介

双端处理是一个本地 Web 工具，用于协助完成标签处理工作。系统提供 A 端操作：

- **A 端**（主操作端）：上传文件、执行处理、处理异常、下载结果、查询货号历史

---

## 启动方法

在 `双端处理/` 目录下执行：

```
python server.py
```

启动后终端会打印局域网访问地址，例如：

```
服务已启动，A机浏览器访问：http://192.168.1.100:5000
```

- A 端访问：`http://<IP>:5000/`

---

## 文件准备

处理前需准备以下三类文件：

| 文件类型 | 格式 | 识别方式 |
|--------|------|---------|
| 扫描文件 | `.xlsx` | 任意名称 |
| 系统文件 | `.csv` | 文件名必须包含 `stockpile` |
| 模板文件 | `.csv` | 文件名必须包含 `模板` |

**说明：**
- 可同时上传多个扫描文件，系统自动合并处理
- 扫描文件取第一列：字母开头识别为库位，数字开头识别为条码
- 模板文件只保留列结构，用于决定输出 CSV 的列顺序

---

## A 端操作流程

### 第一步：上传文件

1. 打开 A 端页面 `/`
2. 将文件拖入上传区，或点击选择
3. 点击「上传文件」按钮

### 第二步：开始处理

点击「▶ 开始处理」。系统会依次执行三个阶段：

**阶段 1：** 解析扫描文件，识别条码与库位

**阶段 2：** 与系统 stockpile 文件匹配，识别新品

**阶段 3：** 按模板生成导入 CSV，归档原始扫描文件

---

## 异常处理

处理过程中可能出现以下三种暂停情况：

### 库位格式异常（阶段 1）

**触发条件：** 扫描文件中识别到的库位不符合「字母+数字」格式（如 A01、X12）

**处理方式：**
1. 左侧面板切换为「库位格式校正」，列出所有格式异常的库位
2. 对每个库位点击「校正」，输入正确库位名（如将 `A` 改为 `A01`）
3. 全部处理完成后，点击「▶ 继续处理（库位已修正）」

### 条码异常（阶段 1）

**触发条件：** 部分条码长度偏离主体长度（统计方法：IQR 检测）

**处理方式：**

| 操作 | 说明 |
|------|------|
| 校正 | 输入正确条码，替换异常条码 |
| 删除 | 从映射中删除该条码 |
| 忽略 | 保留原值不作处理（仅本地标记） |

处理完成后点击「▶ 继续处理（异常已处理）」

### 新品条码（阶段 2）

**触发条件：** 扫描到的条码在 stockpile 系统文件中不存在

**处理方式：** 查看新品列表，确认后点击「▶ 继续处理（新条码已确认）」

---

## 重复条码规则

同一条码在扫描文件中出现多次时：

- **库位相同：** 系统自动去重，只计一次
- **库位不同：** 系统自动将两个库位合并，以 `/` 分隔写入结果（如 `A01/B02`）

终端日志中会出现 `[重复条码-多库位]` 提示。

---

## 输出结果

处理完成后：

- 点击「⬇ 下载结果」下载 ZIP 压缩包
- 压缩包中包含：处理后的模板 CSV + 原始扫描文件整理版

输出目录命名规则：`{员工名}价格标{时间戳}`，例如 `ISLAM价格标20260411112637`

**归档规则：**
- 扫描文件 → 移动到 `垃圾桶/`
- stockpile 系统文件 → 移动到 `垃圾桶/`
- 模板文件 → 保留在 `input/`（可复用）

---

## 重复条码检查

点击左侧导航「🔍 重复条码检查」，上传任意 `.xlsx / .xls / .csv` 文件：

- 检查第一列是否存在重复值
- 返回重复值、出现次数和行号列表
- 适合在正式处理前排查扫描文件中的条码重复

---

## 📜 货号历史

点击左侧导航「📜 货号历史」：

- 输入条码或型号（精确匹配），查询当前主表中的库位、状态、来源等信息
- 显示所有历史变更记录，按时间倒序排列
- 5 秒内的多次变更会聚合为一次操作显示
- 只读功能，不可编辑
- 如需纠错，请通过现有的 import 或手动修正流程进行

---

## 互传功能（A 端 ↔ A 端，双机协作）

页面右侧常驻互传面板，支持多机器的 A 端之间协作：

**文件互传：**
- A 端可拖入文件发送给其他 A 端机器
- 其他机器的 A 端可在「文件」区查看并下载

**文本互传：**
- 多机器的 A 端均可发送短消息
- 消息带发送方标识，可单独删除

## 开发：如何加新 schema 字段

数据层用 SQLAlchemy ORM + Alembic 管理（阶段 1.3 起）。schema 单源在 `models.py`。

**步骤**：

1. 改 `models.py` 对应类（如 `Stockpile`）加新列：
   ```python
   class Stockpile(Base):
       ...
       new_field: Mapped[Optional[str]] = mapped_column(Text)
   ```

2. 生成迁移：
   ```bash
   alembic revision --autogenerate -m "add stockpile.new_field"
   ```

3. 检查 `alembic/versions/<新文件>.py` 内容是否符合预期，必要时手工调整（SQLite ALTER 限制可能需要 `op.batch_alter_table`）。

4. 应用到本地 DB：
   ```bash
   alembic upgrade head
   ```

5. 跑 `pytest` 验证零回归。

**回滚**：`alembic downgrade -1`

**当前 head**：`alembic current` 查看；首次 baseline 是空 revision `cf04ed0496f7`。

## 常见问题

**Q：页面提示「没有收到文件」**
上传时文件列表为空，请重新选择文件后再上传。

**Q：提示找不到 stockpile 或模板文件**
检查文件名：系统文件必须包含 `stockpile`，模板文件必须包含 `模板`。

**Q：处理中断后无法继续**
不要手动删除 `input/` 下的 `_temp_mapping.json` 和 `_temp_results.json`，这两个文件用于阶段间传递中间结果。

**Q：CSV 显示乱码**
模板和系统 CSV 优先按 UTF-8 读取，失败时回退 GBK，通常可自动兼容。若仍有问题，检查终端编码：

```powershell
[Console]::OutputEncoding = [System.Text.Encoding]::UTF8
```
