## 摘要

本文针对中小零售企业（特别是希腊本地化场景）面临的多源 ERP 数据接入复杂、SKU 数量多、销售数据稀疏、库存与销售决策缺乏数据支撑等问题，设计并实现了一个完整的全链路进销存管理与销售预测系统。系统覆盖从扫描标签处理、库存主档管理、客户与供应商分类、ERP 数据清洗与导入、SKU 销售形态分类、时序预测算法实现、滚动回测评估到生产环境部署的全流程。本文核心创新有两点：(1) 基于 4 类生命周期（new / seasonal / declining / stable）与 4 类销售形态（retail_dominant / mixed / wholesale_only / dying）的双正交分类体系，为不同形态 SKU 路由至适当的预测路径；(2) 通过 v1 全量回测的系统性偏差排查发现 CrostonSBA 在已停售 SKU 上的过预测问题，提出基于「最后销售距 as_of ≥ 13 周」的 dying 检测机制，将 dying SKU 路由出时序预测路径，显著降低系统性偏差。系统在一家希腊零售企业的真实 ERP 数据上（1,361,065 条事件、27,340 个活跃 SKU、3 年历史）实测验证。实验结果表明：(1) NaiveSeasonal52W 在 base_demand 视图下是综合最优 baseline（MAPE 中位数 0.85，coverage@p98 = 92.1%）；(2) CrostonSBA 不做 dying 分流时存在 +0.89 的系统性偏差；(3) 完整系统通过 708 个单元测试，并已通过 Docker + Coolify 在 Hetzner 上部署投入实际使用。

**关键词**: 进销存管理；销售预测；间歇需求；Croston-SBA；SKU 分类；滚动回测；Flask；SQLite；Docker

## Abstract

This thesis addresses the comprehensive challenges of small-to-medium retail enterprise management, particularly in the Greek localization context, including heterogeneous ERP data integration, large SKU counts, sparse sales data, and lack of data-driven decision support for inventory and sales operations. We design and implement an end-to-end enterprise management and sales forecasting system covering label scanning workflows, inventory master data management, customer and supplier classification, ERP data cleaning and ingestion, SKU sales-pattern classification, time-series forecasting algorithm implementation, walk-forward backtest evaluation, and production deployment. Two core contributions: (1) a dual-orthogonal SKU classification scheme combining four lifecycle categories (new / seasonal / declining / stable) with four sales-pattern categories (retail_dominant / mixed / wholesale_only / dying), routing each SKU to appropriate forecasting paths; (2) a dying-SKU detection mechanism (last sale ≥ 13 weeks ago), discovered through systematic bias analysis of v1 full backtest, which substantially reduces CrostonSBA's over-forecasting on stopped-sale SKUs. The system is validated on real ERP data from a Greek retailer (1,361,065 events, 27,340 active SKUs, 3-year history). Results: (1) NaiveSeasonal52W is the best overall baseline under base_demand view (median MAPE 0.85, coverage@p98 = 92.1%); (2) CrostonSBA exhibits a +0.89 systematic bias without dying-SKU filtering; (3) the complete system passes 708 unit tests and has been deployed in production via Docker + Coolify on Hetzner.

**Keywords**: Inventory management; Sales forecasting; Intermittent demand; Croston-SBA; SKU classification; Walk-forward backtest; Flask; SQLite; Docker

## 第一章 绪论

### 1.1 研究背景与意义

中小零售企业（Small and Medium-sized Retail Enterprises, SMRE）的运营管理普遍面临数字化程度低、数据散落多端、决策缺乏量化支撑等问题。本论文研究对象为一家希腊本地零售企业（含中国采购视角与希腊本土销售视角），其日常运营需要同时处理以下问题：

1. **标签管理**: 扫码识别 SKU、模板批量生成、异常条码人工修正。错误的标签处理直接影响入库准确性。
2. **库存管理**: 数万 SKU 的库存主档维护、库位管理、人工分级与分类标签。
3. **客户分类**: 区分老外（foreign）客户、中国客户、混合客户与未知客户，为差异化营销提供基础。
4. **进销存数据采集**: 从 ERP（Boson）系统抓取销售与采购明细，清洗噪音后入库。
5. **销售预测与补货决策**: 在数据稀疏（空周比例中位数 73%-90%）、SKU 异质（零售 vs 批发）、生命周期不一（新品 vs 稳态 vs 停售）的复杂场景下做精度可量化的预测。

大型零售企业有条件应用商业 ERP（SAP、Oracle 等）配合机器学习与深度学习模型，但中小企业受限于数据量、IT 预算与人员配比，需要轻量化、可自部署、可解释的系统。本文系统正是面向这类场景设计。

### 1.2 国内外研究现状

**进销存系统设计**: 早期文献多聚焦企业级 ERP（如 SAP R/3）的二次开发，对小型企业适用性差。Flask + SQLite 的轻量化进销存设计模式 [11] 提供了「小而完整」的参考样板。本文系统在此思路上扩展，加入了 ERP 异构数据接入、自动 SKU 分类、时序预测等模块。

**间歇需求预测**: 最早的间歇需求专用算法由 Croston (1972) 提出 [1]，将销量序列拆解为非零需求大小与需求间隔两个子序列分别做指数平滑。Syntetos & Boylan (2005) 指出 Croston 原方法存在系统性偏差，提出 SBA 修正项 `(1 - α/2)` [2]，被广泛认为是间歇需求预测的强基线。Teunter, Syntetos & Babai (2011) 进一步提出 TSB 方法 [3]，用「需求概率」替代「需求间隔」，对生命末期 SKU 表现更优。

**预测精度评估**: 对间歇需求，传统的 MAPE 在实际销量为 0 时不可定义。Hyndman & Koehler (2006) 提出 MASE [4]，以 lag-1 naive 模型的 MAE 作为分母进行缩放，对零值稳健且具有跨序列可比性。

**SKU 分类**: ABC-XYZ 分类法 [5]、生命周期模型 [6]、需求模式分类（smooth / intermittent / lumpy / erratic）[7] 等均提出基于 SKU 特征进行预测方法分流的思路，但鲜有覆盖到「最近无销售但历史活跃」的「停售」状态。本文提出的 `dying` 状态正是为补充这一空缺。

**SaaS 化部署**: Docker + Coolify 等开源 PaaS 让小型 Web 应用的运维门槛大幅降低 [12]。本文系统部署正是这种 self-hosting 思路的实践。

## 第二章 相关理论与技术

### 2.1 时间序列预测理论

时间序列预测的基本任务是给定历史观测 $\{y_1, y_2, \ldots, y_T\}$，预测未来 $h$ 步 $\hat{y}_{T+1}, \ldots, \hat{y}_{T+h}$。本文中所有预测同时给出均值 $\mu$ 与分位数 $p_{50}$、$p_{98}$，便于业务侧做缺货风险评估。

按底层假设分类：

- **平稳模型**: Naive、Moving Average、Exponential Smoothing
- **趋势模型**: Linear Regression on Time、Holt's method
- **季节性模型**: SARIMA、Holt-Winters
- **间歇需求模型**: Croston、SBA、TSB
- **机器学习模型**: XGBoost、LSTM、N-BEATS、Prophet（本文范畴外）

#### 2.1.1 Croston 算法与 SBA 修正

设原始序列 $\{y_t\}$，其中部分 $y_t = 0$。Croston 拆为两个子序列：需求大小 $z_t$（非零观测值）与需求间隔 $p_t$（相邻非零间隔）。对两者分别做指数平滑（平滑系数 $\alpha \in (0, 1)$）：

$$
\hat{z}_{t+1} = \alpha y_t + (1 - \alpha) \hat{z}_t \quad \text{(仅在非零事件后更新)}
$$

$$
\hat{p}_{t+1} = \alpha k_t + (1 - \alpha) \hat{p}_t \quad \text{(仅在非零事件后更新)}
$$

单期需求预测：$\hat{y}_t^{\text{Croston}} = \hat{z}_t / \hat{p}_t$。

SBA 修正：$\hat{y}_t^{\text{SBA}} = \left(1 - \alpha/2\right) \cdot \hat{z}_t / \hat{p}_t$。当 $\alpha = 0.1$ 时修正因子 0.95。

#### 2.1.2 预测精度评估指标

**MAPE**: $\text{MAPE} = \frac{1}{n} \sum |y_t - \hat{y}_t| / y_t$。直观但 $y_t = 0$ 时不可定义。

**MASE**: $\text{MASE} = \text{MAE}(\text{model}) / \text{MAE}(\text{lag-1 naive})$。对零值稳健，跨序列可比；< 1 优于 naive。

**Bias**: $\text{Bias} = \frac{1}{n} \sum (\hat{y}_t - y_t)$。带符号；正值表示系统性高估。

**Coverage@p98**: 实际销量 $\le$ 预测 98 分位上界的占比。

### 2.2 进销存信息系统理论

进销存（Purchase-Sale-Inventory, PSI）系统的核心数据模型包含：

- **主档（Master Data）**: 商品（SKU）、客户、供应商、库位
- **事件（Transaction Data）**: 销售明细、采购明细、入库、出库、调拨
- **状态（Status Data）**: 库存余量、库存周转、客户欠款

PSI 系统通常需要支持的核心功能 [13]：

1. **数据采集**: ERP / POS / Excel 等多源数据接入
2. **主档维护**: SKU 增删改、合并去重、字段补全
3. **事件录入**: 手工录入 / 批量导入 / API 接口
4. **库存计算**: 期初 + 进 - 出 = 期末
5. **报表查询**: 库存日报、销售周报、月度汇总、客户对账
6. **决策支持**: 滞销品识别、补货建议、采购预测

本文系统在传统 PSI 基础上加入：(1) 自动 SKU 分类（生命周期 + 销售形态双正交），(2) 时序销售预测，(3) 回测评估闭环。

### 4.4 SKU 4 类生命周期分类

#### 4.4.1 4 类定义

| 类别 | 规则 |
|---|---|
| `new` | 首次销售距 as_of < 4 周 |
| `seasonal` | 周销量序列长度 ≥ 52 周 + 滞后 52 周 ACF > 0.5 + 至少 2 完整年度重复峰季 |
| `declining` | 最近 4 周斜率 < 0 + 上一季度销量比再上一季度跌 ≥ 30% |
| `stable` | 寿命 ≥ 26 周 + 销售周占比 ≥ 50% + 最近 12 周斜率 ∈ [-10%, +10%] |
| `unclassified` | 都不满足 |

判定优先级：`new > seasonal > declining > stable > unclassified`。

#### 4.4.2 季节性判定（ACF52 + 峰季对照）

判定 SKU 是否季节性的核心是计算滞后 52 周的自相关系数（ACF）：

$$
\text{ACF}(52) = \frac{\frac{1}{N - 52} \sum_{t = 52}^{N - 1} (y_t - \bar{y})(y_{t-52} - \bar{y})}{\frac{1}{N} \sum_{t=0}^{N-1} (y_t - \bar{y})^2}
$$

使用 unbiased 估计（分子除以 $N - 52$ 而非 $N$）原因：biased 版本在 lag 接近 N/2 处会被大 N 平均稀释。N=70 时纯正弦序列只能得 ~0.26，而 unbiased 接近 0.5+。

阈值 ACF > 0.5 通过后进入第二步「峰季对照」：取序列 80% 分位以上为峰周，看前后两个 52 周窗口的峰周重叠率，≥ 30% 视为真季节性（防止偶然 ACF 高的伪季节）。

#### 4.4.3 衰退判定

最近 4 周线性回归斜率 < 0 + 上一季度（90 天）销量比再上一季度（90-180 天）跌 ≥ 30%。

### 5.3 退货归并机制

#### 5.3.1 问题

ERP 中退货事件以**负数 qty** 表示，通过 `document_no` 与原销售单关联。直接 `SUM(qty)` 周聚合会让退货在不同周抵消进需求。例如：

| event_at | qty | document_no |
|---|---|---|
| 2026-05-06 | +10 | D1 |
| 2026-05-13 | -3 | D1 |

朴素聚合：第一周 +10，第二周 -3。业务真相：D1 净需求 7，全部归到原下单周。

#### 5.3.2 算法

`forecast_data.weekly_demand_series`:

```
1. 拉窗口内 sale 事件
2. 按 document_no 分组 (None 按事件主键独立)
3. 对每组求 net = SUM(qty):
   - net > 0: 挂到组内最早 event_at 所属周
   - net ≤ 0: 丢弃 (孤儿退货或完全冲销)
4. 空周补 0
```

#### 5.3.3 边界处理

- **孤儿退货**: 原单在窗口外。算法**丢弃**避免落负。
- **跨周退货**: 净量挂**最早事件**所属周。
- **NULL document_no**: 按事件主键独立成组。

#### 5.3.4 测试

12 个 unittest 覆盖（`tests/test_forecast_data.py`）：空数据 / 窗口边界 / 同 doc 同周净抵 / 跨周净抵 / 孤儿退货丢弃 / 跨 doc 累加 / 空周补 0 / NULL doc / 完全冲销 / purchase 排除 / 其他 barcode 排除。

#### 5.4.2 判定规则

| 类别 | 判定条件 | 处理 |
|---|---|---|

| `mixed` | 其余 | 进时序预测 + 客户类型过滤 |

判定优先级：`unclassified > dying > wholesale_only > retail/mixed`。

#### 5.4.3 阈值选定

| mixed | 312 | 1.0% |

#### 5.4.4 dying 检测的引入

dying 阈值 N=13 周（约 1 季度）经 spike `_scratch/spike_dying_threshold.py` 选定。理由：(1) 业务直观，(2) 与新品判定 90 天对称，(3) 全量 46.3% SKU 被标 dying。详见 §6.5 案例分析的发现过程。

## 第六章 销售预测算法与回测分析（作者 A 主笔）

### 6.1 ForecastModel Protocol

为统一抽象不同算法接口，使用 Python `typing.Protocol`：

```python
class ForecastModel(Protocol):
    name: str
    def fit(self, history: list[float]) -> None: ...
    def predict(self, steps: int = 1) -> ForecastDist: ...
```

`ForecastDist` 是 dataclass，含 `mu` / `sigma` / `p50` / `p98`。`p98` 由 $\hat{\mu} + 2.054 \cdot \hat{\sigma}$ 近似（$z_{0.98} = 2.054$），约束 $\ge 0$。

### 6.2 四个 Baseline 算法

#### 6.2.1 NaiveMean4W

```python
mu = mean(history[-4:])
sigma = std(history[-4:])
```

适用稳态需求；最简单的强基线。

#### 6.2.2 NaiveSeasonal52W

```python
if len(history) < 52:
    mu = mean(history)
else:
    mu = history[-52]
    sigma = std(history[i] - history[i-52] for i ≥ 52)
```

适用周期 52 周季节性 SKU；不足 52 周退化为全历史均值。

#### 6.2.3 LinearTrend12W

```python
window = history[-12:]
if len(window) < 4:
    mu = mean(window)
else:
    slope, intercept = polyfit(arange(n), window, 1)
    mu = slope * n + intercept
    sigma = std(残差)
```

适用线性趋势 SKU；实测在间歇序列上表现最差（详见 §6.4）。

#### 6.2.4 CrostonSBA

```python
def fit(history, alpha=0.1):
    nz_idx = [i for i, v in enumerate(history) if v > 0]
    size = history[nz_idx[0]]
    interval = nz_idx[0] + 1
    prev = nz_idx[0]
    for idx in nz_idx[1:]:
        gap = idx - prev
        obs = history[idx]
        size = alpha * obs + (1 - alpha) * size
        interval = alpha * gap + (1 - alpha) * interval
        prev = idx
    mu = (size / interval) * (1 - alpha / 2)
    sigma = std(history - mu)
```

适用间歇需求 SKU。$\alpha = 0.1$ 经典经验值。

### 6.5 CrostonSBA 系统性 Bias 案例分析（本文核心创新）

#### 6.5.1 排查

通过 spike `_scratch/spike_croston_bias.py` 提取 v1 CrostonSBA Bias 最高的 5 个 SKU。详细看 Top 3：

| SKU | 最后销售距 as_of | SBA size/interval | predicted | test actuals |
|---|---|---|---|---|
| 8299979002791 | 63 周 | 1503 / 3.48 = 432 | 410.6 | [0, 0, 0, 0] |
| 8432011600443 | 13 周 | 82.8 / 4.36 = 19.0 | 18.0 | [0, 0, 0, 0] |
| 5828079245994 | 18 周 | 48.6 / 5.97 = 8.1 | 7.7 | [0, 0, 0, 0] |

**关键发现**：3 个 over-forecast SKU 都是 **stop-sale**（最后销售在很久以前），但 SBA 的 EWMA size 与 interval 仍按全部历史指数平滑，**没有「最后非零事件距 as_of 多久」这一信号**，导致 mu 仍为正值。SBA 在「稳态间歇」假设下设计，这是其在 dying SKU 上系统性高估的根因。

#### 6.5.2 修复方案对比

| 方案 | 改动位置 | 优缺点 |
|---|---|---|
| A. 加 dying SKU 状态 | 数据层（categorizer） | ✅ 治本；所有 baseline 都受益；未来引入算法也免疫 |
| B. baseline 内嵌 dying 检测 | 各 baseline 的 fit() | ❌ 治标；多算法重复实现；耦合 |
| C. SBA 换 TSB | 算法 | ❌ 只修复 SBA |

本文选 **方案 A**，从数据层而非算法层解决，所有 baseline 都受益。

#### 6.5.3 修复实现与验证

如 §5.4.4，在 `classify_sku_type` 前置 dying 判定。改动文件：`categorizer.py`、`forecast_data.py`、`backtest_service.py`，新增 6 个 unittest。

**深入诊断**:

这暗示 retail_dominant 池内存在「低频但未完全停售」的 SKU —— 例如最后销售在 13 周内（不满足 dying 阈值）但近期销售频率明显下降的 SKU。CrostonSBA 的 EWMA size 在 α=0.1 时衰减太慢，对这类 SKU 仍会高估。

**进一步修复方向** (留待后续工作):

1. **细化 dying 阈值**: N=13 改为 N=8 周或加分级状态（如「decelerating」），实测见 §6.5.5
2. **调整 α**: 提高至 0.2-0.3 让 CrostonSBA 对近期变化更敏感
3. **算法替换**: 用 TSB（Teunter-Syntetos-Babai）替代 SBA。TSB 的「需求概率」每周都更新（即使零值周也更新），对衰减更敏感。是答辩前最值得做的对照实验。

### A.1 退货归并 — 作者 A（`forecast_data.weekly_demand_series`）

```python
def weekly_demand_series(barcode, end_date, weeks, session=None):
    end_monday = _monday(end_date)
    week_starts = [end_monday - timedelta(days=7 * (weeks - 1 - i))
                   for i in range(weeks)]
    series = {w: 0 for w in week_starts}
    rows = fetch_sale_events(barcode, window_start, window_end)
    buckets = defaultdict(list)
    for event_at, qty, doc_no, ev_id in rows:
        key = doc_no if doc_no else f"__null__{ev_id}"
        buckets[key].append((parse_date(event_at), qty))
    for items in buckets.values():
        net = sum(q for _, q in items)
        if net <= 0:
            continue
        earliest_week = _monday(min(d for d, _ in items))
        if earliest_week in series:
            series[earliest_week] += net
    return series
```

```python
def _acf_at_lag(series, lag):
    """unbiased 自相关系数估计."""
    n = len(series)
    if lag <= 0 or lag >= n:
        return None
    y = np.asarray(series, dtype=float)
    mean = y.mean()
    centered = y - mean
    var_mean = (centered * centered).sum() / n
    if var_mean <= 0:
        return None
    overlap = n - lag
    cov_mean = (centered[lag:] * centered[:-lag]).sum() / overlap
    return cov_mean / var_mean
```

### A.4 CrostonSBA — 作者 A（`backtest_service.CrostonSBA.fit`）

```python
def fit(self, history):
    if not history:
        self._dist = _zero_dist()
        return
    nz_idx = [i for i, v in enumerate(history) if v > 0]
    if not nz_idx:
        self._dist = _zero_dist()
        return
    size = history[nz_idx[0]]
    interval = nz_idx[0] + 1
    prev = nz_idx[0]
    for idx in nz_idx[1:]:
        gap = idx - prev
        obs = history[idx]
        size = self.alpha * obs + (1 - self.alpha) * size
        interval = self.alpha * gap + (1 - self.alpha) * interval
        prev = idx
    mu = (size / interval) * (1 - self.alpha / 2)  # SBA 修正
    sigma = np.std(np.asarray(history) - mu)
    self._dist = _dist_from_mu_sigma(mu, sigma)
```

def detect_barcode_outliers(barcodes, min_len=8, max_len=14):
    """检测扫描条码中长度异常的 (拼接 / 断行 / 扫描错位)."""
