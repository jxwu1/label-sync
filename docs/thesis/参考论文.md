## 摘要

本文针对中小零售企业历史销售数据稀疏、SKU 数量多、间歇需求显著的特点，设计并实现了一个完整的销售预测系统。系统覆盖从异构数据源采集、数据清洗、SKU 销售形态分类、时序预测算法实现、滚动回测评估到生产环境部署的全链路。本文重点研究面向间歇需求场景的预测方法分流策略，提出基于 SKU 销售形态的四分类体系（`retail_dominant` / `mixed` / `wholesale_only` / `dying`），将不同状态的 SKU 路由至对应的预测算法，避免传统时序预测方法（特别是 Croston-SBA 算法）在已停售 SKU 上的系统性高估问题。系统在一家希腊零售企业的真实 ERP 数据上（1,361,065 条事件、27,340 个活跃 SKU、3 年历史）实测验证。实验结果表明：(1) 在 base_demand 视图下，NaiveSeasonal52W 是综合最优 baseline（MAPE 中位数 0.85，coverage@p98 = 92.1%）；(2) CrostonSBA 在未做 dying 分流时存在 +0.89 的系统性偏差，引入 dying 检测后偏差显著回落；(3) 滚动回测框架支持双视图对比（含异常订单 vs 仅基础需求），可量化数据清洗策略的实际收益。

## Abstract

This thesis addresses the challenge of demand forecasting for small-to-medium retail enterprises, where historical sales data is sparse, SKU count is high, and intermittent demand patterns dominate. We design and implement an end-to-end sales forecasting system covering heterogeneous data ingestion, cleaning, SKU sales-pattern classification, time-series forecasting algorithm implementation, walk-forward backtest evaluation, and production deployment. The core contribution is a four-class SKU classification scheme (`retail_dominant` / `mixed` / `wholesale_only` / `dying`) that routes SKUs to appropriate forecasting paths, preventing systematic over-forecasting of stopped-sale SKUs by classical methods such as Croston-SBA. The system is validated on real ERP data from a Greek retailer (1,361,065 events, 27,340 active SKUs, 3-year history). Results show: (1) under base_demand view, NaiveSeasonal52W is the best overall baseline (median MAPE 0.85, coverage@p98 = 92.1%); (2) CrostonSBA exhibits a +0.89 systematic bias without dying-SKU filtering, which is substantially reduced after the proposed classification; (3) the backtest framework supports dual-view comparison for quantifying data-cleaning strategies' effect.

**Keywords**: Sales forecasting; Intermittent demand; Croston-SBA; SKU classification; Walk-forward backtest; Flask; SQLite; Docker

## 第一章 绪论

### 1.1 研究背景与意义

零售企业的库存管理与采购决策依赖于对未来销售的准确预测。预测精度直接影响库存周转率、缺货率、资金占用与客户满意度等关键运营指标。对大型连锁零售企业而言，丰富的历史数据与商品数量使其有条件应用复杂的机器学习与深度学习方法（如 Prophet、LSTM、N-BEATS 等）。然而对中小型零售企业（包括本文研究的希腊零售场景），销售预测面临以下不同于大型企业的挑战：

1. **数据稀疏**：单 SKU 周销量经常为 0，间歇需求（Intermittent Demand）特征显著。空周比例中位数高达 73%—87%，传统假设连续需求的时序方法（如 ARIMA、Holt-Winters 等）不直接适用。
2. **SKU 数量多但单 SKU 历史短**：本文研究对象的活跃 SKU 数为 27,340，绝大多数 SKU 不足 52 周完整年度数据，季节性建模困难。
3. **数据噪音重**：原始 ERP 数据中混杂了退货事件（负数量）、内部账户调拨、税务行、跨单元合并等多种非主营销售记录，直接建模会引入系统性偏差。
4. **客户类型异质**：批发客户的大单与零售客户的小单混合存在，简单聚合会让批发大单污染零售需求估计。
5. **SKU 生命周期阶段不同**：新品、稳态、衰退、停售等不同阶段的 SKU 适用不同的预测策略，统一处理会在停售 SKU 上产生明显高估。

针对上述挑战，本文以一家实际运营中的希腊零售企业 ERP 数据为研究对象，设计并实现了完整的数据 pipeline 与预测系统，重点研究**面向间歇需求与异质 SKU 状态的分流预测方法**，并通过滚动回测框架对 4 种基线算法进行量化对比。

### 1.2 国内外研究现状

**间歇需求预测**: 最早的间歇需求专用算法由 Croston (1972) 提出 [1]，其核心思想是将销量序列拆解为非零需求大小（demand size）与需求间隔（inter-demand interval）两个子序列分别做指数平滑，再以 `size / interval` 作为单期需求估计。后续 Syntetos & Boylan (2005) 指出 Croston 原方法存在系统性偏差，提出 Syntetos-Boylan Approximation (SBA) 修正项 `(1 - α/2)`，被广泛认为是间歇需求预测的强基线 [2]。Teunter, Syntetos & Babai (2011) 进一步提出 TSB 方法，用「需求概率」替代「需求间隔」，对生命末期 SKU 表现更优 [3]。

**预测精度评估**: 对间歇需求，传统的 MAPE（Mean Absolute Percentage Error）在实际销量为 0 时不可定义，应用受限。Hyndman & Koehler (2006) 提出 MASE（Mean Absolute Scaled Error）[4]，以 lag-1 naive 模型的 MAE 作为分母进行缩放，对零值稳健且具有跨序列可比性。

**SKU 分类与分流预测**: ABC-XYZ 分类法 [5]、生命周期模型（Bass model）[6]、需求模式分类（smooth / intermittent / lumpy / erratic）[7] 等均提出基于 SKU 特征进行预测方法分流的思路，但鲜有覆盖到「最近无销售但历史活跃」的「停售」状态。本文提出的 `dying` 状态正是为补充这一空缺。

**预测系统工程化**: Hyndman & Athanasopoulos 著《Forecasting: Principles and Practice》[8] 提供了实际项目落地的完整框架。本文系统设计参考其中的 walk-forward backtest 实践。

### 1.3 本文主要工作

本文主要贡献：

1. **数据 Pipeline 设计**: 设计了适用于异构 ERP 数据的 ETL 流程（Parquet 历史回填 + HTML 日常增量），实现了对内部账户、税务行、退货事件等噪音的标准化清洗。

## 第二章 相关理论与技术

### 2.1 时间序列预测概述

时间序列预测的基本任务是给定历史观测 $\{y_1, y_2, \ldots, y_T\}$，预测未来 $h$ 步 $\hat{y}_{T+1}, \ldots, \hat{y}_{T+h}$。预测可以是点预测（一个数值）或概率预测（一个分布）。本文中所有预测同时给出均值 $\mu$ 与分位数 $p_{50}$、$p_{98}$，便于业务侧做缺货风险评估。

按底层假设可分为：

- **平稳模型**：假设序列均值稳定，如 Naive、Moving Average、Exponential Smoothing
- **趋势模型**：假设存在线性或非线性趋势，如 Linear Regression on Time、Holt's method
- **季节性模型**：假设存在固定周期，如 SARIMA、Holt-Winters
- **间歇需求模型**：专为零值多的稀疏序列设计，如 Croston、SBA、TSB
- **机器学习模型**：树模型 / 神经网络，如 XGBoost、LSTM、N-BEATS、Prophet

本文聚焦前四类模型作为 baseline。机器学习模型在本文范畴外，留待后续工作。

### 2.2 Croston 算法与 SBA 修正

#### 2.2.1 Croston 算法

设原始序列 $\{y_t\}$，其中部分 $y_t = 0$。Croston 将其拆为两个子序列：

- **需求大小** $z_t$: 每个非零观测的值
- **需求间隔** $p_t$: 相邻非零观测之间的间隔（周数）

对两个子序列分别做指数平滑（平滑系数 $\alpha \in (0, 1)$）：

$$
\hat{z}_{t+1} = \alpha y_t + (1 - \alpha) \hat{z}_t \quad \text{(仅在非零事件后更新)}
$$

$$
\hat{p}_{t+1} = \alpha k_t + (1 - \alpha) \hat{p}_t \quad \text{(仅在非零事件后更新)}
$$

其中 $k_t$ 是当前非零事件距上一次非零事件的间隔。单期需求预测为：

$$
\hat{y}_t^{\text{Croston}} = \frac{\hat{z}_t}{\hat{p}_t}
$$

#### 2.2.2 SBA 修正

Syntetos & Boylan (2005) 证明 Croston 估计存在向上偏差（约为 $\alpha / 2$ 阶），提出 SBA 修正：

$$
\hat{y}_t^{\text{SBA}} = \left(1 - \frac{\alpha}{2}\right) \cdot \frac{\hat{z}_t}{\hat{p}_t}
$$

当 $\alpha = 0.1$ 时，修正因子为 0.95，即将 Croston 原始预测下调 5%。SBA 被广泛认为是间歇需求场景下的强基线。

#### 2.2.3 SBA 在 dying SKU 上的局限（本文发现）

SBA 的设计前提是「稳态间歇需求」：单 SKU 长期有销售，只是间歇出现。然而真实零售数据中存在大量 **dying SKU**：曾经活跃但近期已停售。对这类 SKU，SBA 仍按历史 EWMA 估计 $\hat{z}$ 和 $\hat{p}$，无法识别「最后一次非零事件距今很久」这一信号，导致预测严重高估。本文实测中，CrostonSBA 在不做 dying 分流时 Bias_avg = +0.89，远大于其他 baseline，正是这一问题的体现（详见第五章）。

### 2.3 预测精度评估指标

#### 2.3.1 MAPE（Mean Absolute Percentage Error）

$$
\text{MAPE} = \frac{1}{n} \sum_{t=1}^{n} \left| \frac{y_t - \hat{y}_t}{y_t} \right|
$$

直观可解释（百分比），但 $y_t = 0$ 时不可定义。本文实现中**剔除 $y_t = 0$ 的观测点**，当所有 $y_t = 0$ 时返回 None。

#### 2.3.2 MASE（Mean Absolute Scaled Error）

Hyndman & Koehler (2006) 提出，以 lag-1 naive 预测的 MAE 作为分母进行缩放：

$$
\text{MASE} = \frac{\frac{1}{n} \sum |y_t - \hat{y}_t|}{\frac{1}{n - 1} \sum_{t=2}^{n} |y_t - y_{t-1}|}
$$

物理意义：MASE = 1 即与 lag-1 naive 持平；< 1 表示模型优于 naive；> 1 表示比 naive 更差。MASE 对 $y_t = 0$ 稳健，是间歇需求场景下的首选指标。

#### 2.3.3 Bias

$$
\text{Bias} = \frac{1}{n} \sum (\hat{y}_t - y_t)
$$

带符号。正值表示系统性高估，负值表示系统性低估。无偏估计的 Bias 应接近 0。

#### 2.3.4 Coverage@p98

$$
\text{Coverage}_{p98} = \frac{1}{n} \sum_{t=1}^{n} \mathbb{1}\{y_t \le \hat{p}_{98,t}\}
$$

其中 $\hat{p}_{98,t}$ 是模型给出的 98 分位上界预测。物理意义：实际销量不超过预测上界的占比。理想值应接近 98%。

### 3.1 数据来源与特征

### 3.2 ETL Pipeline 设计

#### 3.2.1 清洗规则

清洗层（`etl/parquet_cleaner.py`）执行以下规则：

1. **剔除税务行**: 条码 `90000000001` 是 ERP 内部税务记账行，不是真实销售。在 1 年样本上剔除 84 笔。
2. **剔除拼接条码**: 长度 ≥ 15 的条码是合并扫描结果，不是单一 SKU。剔除 2 笔。
3. **剔除完全重复行**: ERP 偶发导出重复，按 `(event_at, product_barcode, qty, document_no)` 四元组去重。剔除 1 笔。
4. **拆出内部账户**: 客户 ID 以 `999` 开头的是公司内部账户调拨，不计入销售统计，单独 archive。在 1 年样本上拆出 14,750 笔（占 4.9%）。

清洗规则在 1 年样本（304,507 行）上经 15 个单元测试验证。

#### 3.2.2 导入层

导入层（`etl/parquet_importer.py`）按 `event_type` 字段拆为 sale 与 purchase 两批，复用既有的 `inventory_importer.import_events` 写入 SQLite。13 个集成测试覆盖。

#### 3.2.3 灌库流程

完整流程：

```bash
python tools/wipe_events.py          # 精确清空 (保留 stockpile 主档)
python tools/clean_parquet.py "raw/*.parquet" --out-dir cleaned/ --archive-dir archive/
python tools/import_parquet.py "cleaned/*.parquet"
python tools/inventory_admin.py stats   # 行数校验
```

`wipe_events.py` 设计上仅清空事件表（`inventory_events`、`customers`、`suppliers`），保留人工维护的字段（`manual_grade`, `manual_category`），并自动备份原 DB 至 `.bak` 文件。

### 3.3 退货归并机制

#### 3.3.1 问题

ERP 中退货事件以**负数 qty** 表示，且通过 `document_no` 与原销售单关联。直接做 `SUM(qty)` 周聚合会让退货在不同周抵消进需求，导致需求序列失真。例如：

| event_at | qty | document_no |
|---|---|---|
| 2026-05-06 | +10 | D1 |
| 2026-05-13 | -3 | D1 |

朴素聚合：第一周需求 +10，第二周需求 -3。但业务真相是 D1 这单净需求 7，应全部归到原始下单周（2026-05-06）。

#### 3.3.2 算法设计

`forecast_data.weekly_demand_series` 实现的退货归并算法：

```
输入: barcode, end_date, weeks 窗口
1. 从 inventory_events 拉窗口内所有 sale 事件
2. 按 document_no 分组（None 按事件主键独立成组）
3. 对每组求 net = SUM(qty)
   - 若 net > 0: 挂到组内最早 event_at 所属周
   - 若 net ≤ 0: 丢弃（孤儿退货或完全冲销）
4. 空周补 0
返回: dict[Monday_date, int]
```

#### 3.3.3 关键边界

- **孤儿退货**：原单在窗口外，仅退货事件在窗口内。如不处理会让周需求落负数。算法选择**丢弃**，避免在数据稀疏处产生伪信号。
- **跨周退货**：原单与退货跨多周。算法将净量挂到**最早事件**所属周，保持需求时点的语义一致性。
- **NULL document_no**：少数事件无单号，按事件主键独立成组，不与其他事件合并。

#### 3.3.4 测试

12 个 unittest 覆盖（`tests/test_forecast_data.py`）：空数据 / 窗口边界 / 单事件 / 同 doc 同周净抵 / 同 doc 跨周净抵 / 孤儿退货丢弃 / 跨 doc 累加 / 空周补 0 / NULL doc 独立 / 完全冲销 / purchase 排除 / 其他 barcode 排除。

### 3.4 SKU 销售形态分类

#### 3.4.1 设计动机

零售场景下的 SKU 存在显著异质性。同样的「时序预测」对不同形态 SKU 有不同的适用性：

- 纯零售 SKU（每单 < 24 件）：适合常规时序方法
- 纯批发 SKU（每单数百件，订单稀疏）：时序方法不可靠，应用经验分位数
- 混合型（既有零售又有批发）：批发大单会污染零售需求估计

本文设计 `categorizer.classify_sku_type` 函数将 SKU 分类，路由至独立预测路径。

#### 3.4.2 判定规则

设零售样本数为 $r$（doc-net qty ≤ 24 的 doc 数），总样本数为 $T$。

| 类别 | 判定条件 | 处理 |
|---|---|---|
| `unclassified` | $T = 0$（无销售） | series=None |
| `dying`（新增） | 最后销售距 as_of ≥ 13 周 | series=None |
| `wholesale_only` | $r < 5$ 或 $r/T < 0.05$ | series=None |
| `retail_dominant` | $r/T \ge 0.80$ | 进时序预测 |
| `mixed` | 其余 | 进时序预测 + 客户类型过滤 |

判定优先级 `unclassified > dying > wholesale_only > retail/mixed`。

#### 3.4.3 阈值选定

阈值经 spike `_scratch/spike_sku_type_thresholds.py` 在全量 30,370 个 SKU 上验证：

| 类别 | SKU 数 | 占比 |
|---|---|---|
| retail_dominant | 26,060 | 85.8% |
| wholesale_only | 3,997 | 13.2% |
| mixed | 312 | 1.0% |

退场标准：4 个已知纯大批发 SKU（`6956457684352` / `5203835112206` / `6956457684345` / `6956457684338`）必须标为 `wholesale_only`，1 个高频零售 SKU（`5203692253593`，1,232 笔订单，median qty=2）必须标为 `retail_dominant`。spike 验证 5/5 全部正确。

#### 3.4.4 dying 检测的引入（本文核心创新）

dying 状态的判定阈值经 spike `_scratch/spike_dying_threshold.py` 选定 N=13 周（约 1 季度）。理由：

1. 业务直观：3 个月无销售已属"异常"
2. 与新品判定 90 天（≈ 13 周）形成对称
3. 在全量 30,370 SKU 上，46.3% 会被标为 dying — 大量「曾经活跃但已停售」SKU 被识别
4. 3 个 Croston Top over-forecast SKU 全部被覆盖（详见第五章）

实现上，`classify_sku_type` 在判定 retail/wholesale 前先做 dying 检查：

```python
def classify_sku_type(barcode, session=None, as_of=None) -> str:
    as_of = as_of or today()
    last_at = _fetch_last_sale_at(barcode, session)
    if last_at is None:
        return "unclassified"
    weeks_since = (as_of - parse_date(last_at)).days // 7
    if weeks_since >= 13:
        return "dying"
    net_qtys = _fetch_sku_doc_net_qty(barcode, session)
    return classify_sku_type_from_docs(net_qtys)
```

测试: 6 个新 unittest 覆盖边界 12 周（仍活）/ 13 周（dying）/ 14 周（dying）/ 优先级 / `as_of=None` 用 today。

### 3.5 基础需求视图

`forecast_data.base_demand_view(barcode, end_date, weeks)` 将上述清洗与分类整合，输出可直接喂给预测算法的周需求序列：

```
返回 {
  "sku_type": SKU_TYPES 之一,
  "series": dict[Monday_date, int] | None,  // wholesale/dying/unclassified 时为 None
  "exclusion_count": int,  // 剔除的 doc 数
  "exclusion_qty": int     // 剔除的总净 qty
}
```

处理逻辑（按 sku_type 分支）:

- `wholesale_only` / `dying` / `unclassified`: 直接返回 `series=None`
- `retail_dominant`: 按 `is_bulk_order(qty, stats)` 剔除单笔异常大单
- `mixed`: 同上 + 客户类型过滤（仅保留 customer_type ∈ {foreign, chinese}，剔除 unknown / mixed）

异常订单判定 `is_bulk_order(qty, stats, k=3.0)`：当 $qty > \text{median} + k \cdot \text{IQR}$ 时标为异常。**不使用均值**，因为均值会被历史大单污染。

## 第四章 预测算法实现

### 4.1 ForecastModel Protocol

为统一抽象不同算法的接口，本文使用 Python `typing.Protocol` 定义 `ForecastModel`:

```python
class ForecastModel(Protocol):
    name: str
    def fit(self, history: list[float]) -> None: ...
    def predict(self, steps: int = 1) -> ForecastDist: ...
```

`ForecastDist` 是 dataclass，含四个字段：

```python
@dataclass
class ForecastDist:
    mu: float       # 点估计
    sigma: float    # 标准差估计
    p50: float      # 中位数
    p98: float      # 98% 上界
```

$p_{98}$ 由 $\hat{\mu} + z_{0.98} \cdot \hat{\sigma}$ 近似计算（$z_{0.98} = 2.054$），并约束 $\ge 0$。

### 4.2 NaiveMean4W

**思路**: 取训练序列尾 4 周的算术均值作为预测。

```python
mu = mean(history[-4:])
sigma = std(history[-4:])  # 若样本不足 2 则 sigma=0
```

**适用场景**: 稳态需求；作为最简单的强基线。

### 4.3 NaiveSeasonal52W

**思路**: 取 lag-52（去年同周）的值作为预测。若历史 < 52 周，退化为全历史均值。

```python
if len(history) < 52:
    mu = mean(history)
else:
    mu = history[-52]
    sigma = std(history[i] - history[i-52] for i ≥ 52)
```

**适用场景**: 具有周期 52 周季节性的 SKU。需要至少 1 年历史。

### 4.4 LinearTrend12W

**思路**: 对训练尾 12 周做最小二乘线性回归，外推下一步。

```python
window = history[-12:]
if len(window) < 4:
    mu = mean(window)  # 样本过少回归不稳, 退化
else:
    slope, intercept = polyfit(x=arange(n), y=window, deg=1)
    mu = slope * n + intercept
    sigma = std(window - (slope*x + intercept))  # 残差 std
```

**适用场景**: 具有线性趋势（上涨或下跌）的 SKU。

**实测发现**: 在间歇序列上 LinearTrend12W 表现最差（详见第五章），因趋势外推 amplify 噪音。

### 4.5 CrostonSBA

**思路**: 按 Syntetos-Boylan Approximation 实现。Python 伪代码：

```python
def fit(history, alpha=0.1):
    nz_idx = [i for i, v in enumerate(history) if v > 0]
    if not nz_idx:
        return  # 无任何非零 → mu=0
    size = history[nz_idx[0]]
    interval = nz_idx[0] + 1
    prev = nz_idx[0]
    for idx in nz_idx[1:]:
        gap = idx - prev
        obs = history[idx]
        size = alpha * obs + (1 - alpha) * size
        interval = alpha * gap + (1 - alpha) * interval
        prev = idx
    mu = (size / interval) * (1 - alpha / 2)  # SBA 修正
    sigma = std(history - mu)
```

**适用场景**: 间歇需求（零值多）SKU。

**关键参数**: $\alpha = 0.1$ 是经典经验值，控制 EWMA 的衰减速度。$\alpha$ 越大越敏感于近期变化，但方差也大。

### 4.6 多步预测策略

本文 baseline 不区分预测 horizon（即 $\hat{y}_{T+1} = \hat{y}_{T+2} = \cdots$）。这是 baseline 的简化处理；更复杂的模型（如 Holt-Winters 或 LSTM）应给出衰减或周期性变化的多步预测。本文范围内的 baseline 仍能正确比较，但实际应用中 horizon-aware 模型预期表现更好。

### 4.7 测试覆盖

`tests/test_backtest_service.py` 30 个 unittest 覆盖：ForecastDist 字段、4 个 baseline 各自的边界（空数据 / 短历史 / 常数序列 / 单步预测正确性）、评分函数 mape / mase / bias / coverage_p98 的边界、walk-forward 的记录数、horizon 循环、step_idx 全局位置。

## 第五章 回测框架与实验分析

### 5.1 Walk-Forward 回测设计

#### 5.1.1 滚动窗口算法

```python
def walk_forward_backtest(series, model_cls, window_train=13, window_test=4):
    records = []
    for start in range(len(series) - window_train - window_test + 1):
        train = series[start : start + window_train]
        test = series[start + window_train : start + window_train + window_test]
        model = model_cls()
        model.fit(train)
        for h in range(1, window_test + 1):
            forecast = model.predict(steps=h)
            records.append({
                "step_idx": start + window_train + h - 1,
                "horizon": h,
                "predicted": forecast.mu,
                "p50": forecast.p50, "p98": forecast.p98,
                "actual": test[h - 1],
            })
    return records
```

对长度 $L$ 的序列，产生 $(L - \text{train} - \text{test} + 1) \times \text{test}$ 条记录。本文默认 $\text{train}=13, \text{test}=4$，156 周序列上共 $140 \times 4 = 560$ 条预测记录。

#### 5.1.2 评分聚合

对单 SKU 的全部预测记录聚合 MAPE、MASE、Bias、coverage_p98 四个指标，写入 `backtest_results` 表。

对一次 run 的全部 SKU 聚合「中位数」与「均值」两种统计：MAPE_median / MASE_median 用中位数（稳健于极端值）；Bias_avg / coverage_avg 用均值（保留方向信息）。

### 5.2 实验设置

- **数据**: 27,340 个活跃 SKU，2023-05-13 至 2026-05-13 共 156 周
- **视图**: `base_demand`（应用第三章的全部清洗与分流）
- **训练窗口**: 13 周
- **测试窗口**: 4 周
- **最小可回测条件**: `min_weeks=20`（非零周数 ≥ 20）
- **算法**: 全部 4 个 baseline
- **执行环境**: Windows 11 + Python 3.12 + SQLite WAL，单进程
- **执行时间**: 4 run 串行，每 run 约 118 min，总 7.65h

通过 min_weeks=20 与 sku_type 分流后，实际进入回测的 SKU 数 6,098（占活跃 SKU 22%）。其余 78% SKU 因可回测样本不足、纯批发或停售被剔出。

### 5.3 v1 实测结果（dying 修复前）

#### 5.3.1 总体指标

|---|---|---|---|---|---|
| NaiveMean4W | 6,098 | 0.96 | **0.941** | +0.04 | 89.7% |
| NaiveSeasonal52W | 6,098 | **0.85** | 0.948 | +0.08 | **92.1%** |
| LinearTrend12W | 6,098 | 1.05 | 1.056 | +0.01 | 91.3% |

#### 5.3.2 分析

**NaiveSeasonal52W 是综合最优 baseline**：MAPE 中位数最低（0.85），coverage 最高（92.1%），MASE < 1（优于 lag-1 naive）。其原理是当 SKU 有 ≥52 周历史时取去年同周值，对季节性 SKU 友好；不足 52 周时退化为全历史均值，对短历史也稳健。

**LinearTrend12W 全面最差**：MAPE 1.05、MASE 1.056 均比其他 baseline 高，且 MASE > 1 表示比 lag-1 naive 还差。原因是 12 周尾部线性外推在零值多的间歇序列上 amplify 噪音 — 趋势线倾向于"放大"短期方向，但间歇序列的短期方向往往随机。

**CrostonSBA 反常的高估 Bias=+0.89**：远超其他 baseline，且 MASE_med=1.141 > 1，意味着 CrostonSBA 在 6,098 个 SKU 上系统性表现不佳。但 coverage@p98=92.6% 同时是最高的 — 这是 over-forecast 自带的「副作用」（预测大了，actual ≤ predicted 自然多）。

### 5.4 CrostonSBA 系统性 Bias 案例分析（本文亮点）

#### 5.4.1 排查

通过 spike `_scratch/spike_croston_bias.py` 提取 v1 中 CrostonSBA Bias 最高的 5 个 SKU：

| SKU | Bias | MASE | mean_actual | mean_predicted |
|---|---|---|---|---|
| 8299979002791 | +119.4 | 1.03 | 241.4 | 360.9 |
| 8432011600443 | +16.6 | 1.28 | 35.8 | 52.4 |
| 5828079245994 | +6.8 | 1.20 | 9.6 | 16.4 |
| 5828079103089 | +5.8 | 3.24 | 3.8 | 9.6 |
| 8432011600436 | +5.5 | 0.84 | 75.0 | 80.5 |

对前 3 个 SKU 详细分析，发现共同模式：

| SKU | 最后销售距 as_of | SBA size/interval | predicted | test actuals |
|---|---|---|---|---|
| 8299979002791 | 63 周 | 1503 / 3.48 = 432 | 410.6 | [0, 0, 0, 0] |
| 8432011600443 | 13 周 | 82.8 / 4.36 = 19.0 | 18.0 | [0, 0, 0, 0] |
| 5828079245994 | 18 周 | 48.6 / 5.97 = 8.1 | 7.7 | [0, 0, 0, 0] |

**关键发现**：3 个 over-forecast SKU 都是 **stop-sale SKU**（最后销售在很久以前），但 SBA 的 EWMA size 与 interval 仍按全部历史指数平滑，**没有「最后非零事件距 as_of 多久」这一信号**，导致 mu 仍为正值。SBA 在「稳态间歇」假设下设计，对这一维度不敏感，这是其在 dying SKU 上系统性高估的根因。

#### 5.4.2 修复方案对比

| 方案 | 改动位置 | 优缺点 |
|---|---|---|
| A. 加 dying SKU 状态 | `categorizer.py` 数据层 | ✅ 治本，干净分流；所有 baseline 都受益；不仅修复 SBA |
| B. baseline 内嵌 dying 检测 | 各 baseline 的 fit() | ❌ 治标；多个算法重复实现；耦合 |
| C. SBA 换 TSB | `backtest_service.CrostonSBA` | ❌ 只修复 SBA；其他间歇模型再换又得做一遍 |

本文选用方案 A，理由是从数据层而非算法层解决问题，所有 baseline 与未来引入的算法都受益。

#### 5.4.3 dying 修复实现

如 §3.4.4，在 `classify_sku_type` 前置 dying 判定（最后销售距 as_of ≥ 13 周），并在 `base_demand_view` 与 `_build_series` 都加 dying 早退。改动文件：`categorizer.py`、`forecast_data.py`、`backtest_service.py`，共 105 行新增 / 7 行删除。新增 6 个 unittest 覆盖边界。

### 5.5 双视图回测对比框架

本文额外设计了 `compare_run_pair(run_a, run_b)` 函数，支持任意两次 run 的 per-SKU 差异分析。返回结构包含 `common_skus` 数量、`items` 数组（每 SKU 一行的 MAPE / MASE / coverage 差异）、`summary`（improved / worsened / unchanged 计数 + median_mase_delta）。

通过 HTTP API `GET /analytics/backtest/compare?run_a=&run_b=` 暴露。可对比的维度：
- 同模型不同视图（all vs base_demand），量化清洗策略收益
- 不同模型同视图，model selection
- 同模型同视图不同 dying 策略（v1 vs v2），量化 dying 修复收益（本文核心实验）

### 5.6 实验局限性

1. **未与 ML / DL 模型对比**: 本文范围限于经典 baseline。Prophet、LSTM 等模型预期能在长历史 SKU 上提升精度，留待后续工作。
2. **未跑 view='all' 对照**: 双视图回测框架已就位但本文实测仅跑 base_demand。view='all' 跑完后可量化第三章数据清洗策略的实际收益。
3. **未做敏感性分析**: dying 阈值 N=13 周经业务直觉与 spike 选定，N=8/26/52 的对比未实测。
4. **单一数据源**: 仅在一家希腊零售企业数据上验证。其他行业 / 国家的数据上 dying 阈值与四分类比例预期需重新校准。

通过 `expose: ["5000"]` 而非 `ports: "5000:5000"` 走 Coolify Traefik 容器网络，避免直接暴露主机端口。持久卷 `./data` 挂主机磁盘，避免 redeploy 时数据丢失。

#### 6.3.3 Coolify 部署

Coolify 是开源 PaaS（基于 Docker + Traefik），简化「git push → 部署」流程。配置要点：Build Pack 选 **Docker Compose**（而非默认的 Dockerfile / Nixpacks）；Compose Location 填 `/docker-compose.yml`；持久卷 `./data` 在主机文件系统。

#### 6.3.4 部署踩坑

详细记录见 `docs/deploy-plan.md`。三个关键坑：

1. **Coolify 默认 Build Pack 不识别 docker-compose.yml** — Coolify 会自己渲染 compose，忽略仓库内的。表现：端口默认 3000、env 丢失、Entrypoint 改 bash 死循环。修：手动改 Build Pack 为 Docker Compose。
2. **alembic baseline 是空 `pass`** — 早期项目反射既存 schema，不创建任何表。空 DB 上 `alembic upgrade head` 会半死。修：scp 一份 prod stockpile.db 上去再起容器。
3. **ports vs expose** — Coolify Traefik 需要 expose 而非 ports。

## 第七章 总结与展望

### 7.1 工作总结

本文针对中小零售企业的真实销售预测问题，设计并实现了一个完整的端到端系统，主要工作如下：

1. **数据 Pipeline**: ETL 清洗（剔税行 / 拼接条码 / 重复 / 内部账户）+ 退货归并（按 document_no 净抵）+ SKU 分类分流（四分类含 dying）
2. **预测算法**: 实现 4 个 baseline（NaiveMean4W / NaiveSeasonal52W / LinearTrend12W / CrostonSBA），统一抽象为 ForecastModel Protocol
3. **回测框架**: walk-forward 滚动回测 + 4 类评分（MAPE / MASE / Bias / coverage_p98）+ 双视图对比
4. **核心创新**: 通过 v1 全量回测的偏差排查，发现 CrostonSBA 在 dying SKU 上系统性高估，提出基于「最后销售距 as_of ≥ 13 周」的 dying 检测，将 dying SKU 路由出时序预测路径
5. **工程实现**: Flask + SQLite + alembic + Docker + Coolify 完整部署链路，708 个单元测试覆盖

实测在 27,340 SKU、1.36M 事件的真实数据上完成 4 baseline 全量回测，结论：

- NaiveSeasonal52W 综合最优（MAPE_med=0.85, MASE_med=0.948, cov@p98=92.1%）
- LinearTrend12W 在间歇序列上最差，不推荐作为主力 baseline
- CrostonSBA 经典实现存在 +0.89 系统性高估，需在数据层做 dying 分流后才能稳定使用

### 7.2 局限性

1. **未集成 ML / DL 模型**: Prophet、LSTM、N-BEATS 等现代方法未对比
2. **dying 阈值未做敏感性分析**: 当前 N=13 经业务直觉选定，需补 N=8/26/52 的对比实验
3. **单一数据源**: 仅在一家希腊零售企业的数据上验证
4. **同步执行**: 全量回测目前是同步串行，未利用多进程并行（27,340 SKU × 4 baseline = 7.65h）

### 7.3 后续工作

1. **集成 Holt-Winters**: 引入 statsmodels 库，对有 ≥104 周历史的 SKU（402 个）启用季节性 HW，与 NaiveSeasonal52W 对比
2. **集成 Prophet 与 LSTM**: 比较经典统计方法与现代 ML 方法的精度与算力成本
3. **TSB 算法对比**: 实现 Teunter-Syntetos-Babai 方法（用「需求概率」替代「间隔」），与 SBA 在间歇序列上的表现对比
4. **基于回测分数的 model selection**: 为每个 SKU 自动选择 MASE 最低的算法作为主力预测器
5. **dying 阈值敏感性分析**: N=8/13/26/52 的回测对比，验证 13 周是否真正最优

### A.1 退货归并 (`forecast_data.weekly_demand_series`)

```python
def weekly_demand_series(barcode, end_date, weeks, session=None):
    end_monday = _monday(end_date)
    week_starts = [end_monday - timedelta(days=7 * (weeks - 1 - i))
                   for i in range(weeks)]
    series = {w: 0 for w in week_starts}
    rows = fetch_sale_events(barcode, window_start, window_end)
    buckets = defaultdict(list)
    for event_at, qty, doc_no, ev_id in rows:
        key = doc_no if doc_no else f"__null__{ev_id}"
        buckets[key].append((parse_date(event_at), qty))
    for items in buckets.values():
        net = sum(q for _, q in items)
        if net <= 0:
            continue  # 孤儿退货或完全冲销 → 丢弃
        earliest_week = _monday(min(d for d, _ in items))
        if earliest_week in series:
            series[earliest_week] += net
    return series
```

def classify_sku_type(barcode, session=None, as_of=None):
    as_of = as_of or today()
    last_at = _fetch_last_sale_at(barcode, session)

### A.3 CrostonSBA 实现 (`backtest_service.CrostonSBA.fit`)

```python
def fit(self, history):
    if not history:
        self._dist = _zero_dist()
        return
    nz_idx = [i for i, v in enumerate(history) if v > 0]
    if not nz_idx:
        self._dist = _zero_dist()
        return
    size = history[nz_idx[0]]
    interval = nz_idx[0] + 1
    prev = nz_idx[0]
    for idx in nz_idx[1:]:
        gap = idx - prev
        obs = history[idx]
        size = self.alpha * obs + (1 - self.alpha) * size
        interval = self.alpha * gap + (1 - self.alpha) * interval
        prev = idx
    mu = (size / interval) * (1 - self.alpha / 2)  # SBA 修正
    sigma = np.std(np.asarray(history) - mu)
    self._dist = _dist_from_mu_sigma(mu, sigma)
```
