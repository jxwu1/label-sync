## Phase 1 (2026-06): SQLite → PostgreSQL 迁移

### 故事背景

「现在 SQLite 跑得挺好，为什么要迁？」—— 答辩评委可能这么问。论文 §2.4 给的理由是「数据量小、单写源、运维简单」，这是真话，但答辩讲完后下一句通常是「以后呢」。

「以后」可能发生的事：
1. 客户扩到 2-3 家公司（每家一个 SKU 池），需要 schema 隔离 + 跨公司聚合
2. 后端 worker 数从 1 增到多个（gunicorn `-w 4`），SQLite 写锁打架
3. 引入 cron 任务跑夜间 backtest / 报表，跟前端读冲突
4. 加 ML 模型训练管道，可能需要 R 或 Spark 读 DB

任一情况发生时迁 PG 都不可避免。论文「为支持未来扩展，本系统设计了 SQLite → PostgreSQL 迁移路径」比临时迁要扎实得多。

### 主要工作

#### 1.1 设计阶段（3 天）

- 选 PG 容器（`postgres:16-alpine`）vs 托管（Hetzner 没有 RDS）→ 选容器
- 选连接池（pgbouncer vs SQLAlchemy 内置）→ 简单起见用 SQLAlchemy 内置 pool
- 决定数据迁移工具（pgloader vs 手动 dump/load）→ pgloader（已支持 SQLite → PG）
- 决定测试策略（testcontainers vs 保留 SQLite for tests）→ **保留 SQLite for tests + 加少量 PG smoke tests**

#### 1.2 代码改造（5-7 天）

逐文件审查 SQLite-only 语法：

| 文件 | 问题 | 修改 |
|---|---|---|
| forecast_data.py | 无 | 无 |
| backtest_service.py | 无（NumPy 纯计算） | 无 |
| categorizer.py | `_fetch_last_sale_at` 用 `func.max` 跨 DB | 无 |
| analytics_service.py | 可能用 strftime | grep 后改 `EXTRACT(WEEK FROM ...)` 等 |
| stockpile_db.py | PRAGMA WAL（SQLite 专属） | 加 `if dialect=='sqlite'` 守护 |
| spike_*.py | `'__null__' || ie.id` 用 SQLite || 拼接 | PG 也支持 `||` 但需 cast → `'__null__' || ie.id::text` |
| alembic versions | `server_default=text("(datetime('now','localtime'))")` | 改 `CURRENT_TIMESTAMP` 或加 dialect 分支 |
| alembic env.py | DB URL 写死 sqlite:// | 走 env var `DATABASE_URL` |
| models.py | DB URL | 同上 |
| config.py | `stockpile_db` path 字段 | 加 `database_url` 字段 |
| Dockerfile / compose | 单容器 | 加 postgres service + depends_on |
| requirements.txt | - | 加 psycopg2-binary |

#### 1.3 数据迁移（1-2 天）

```bash
# 1. 起 PG 容器
docker compose up -d postgres

# 2. 用 pgloader 把 SQLite 全部表迁到 PG
docker run --rm --network=label-sync_default \
  -v $(pwd)/stockpile.db:/data/stockpile.db \
  dimitri/pgloader pgloader sqlite:///data/stockpile.db \
  postgresql://label_sync:pass@postgres/label_sync

# 3. 跑 alembic upgrade head (校验 schema 对齐)
docker compose exec label-sync alembic upgrade head

# 4. sanity check: row counts 对得上, 抽样查询结果一致
```

#### 1.4 测试改造（2 天）

`tests/` 已 708 测试，目标是**继续用 SQLite + 加少量 PG 集成测试**：
- pure 函数测试不变（无 DB 依赖）
- DB-bound 测试保留 SQLite mock（速度快）
- 新增 `tests/test_pg_integration.py` 跑几个关键 path（alembic upgrade / 简单 CRUD / 复杂 query）用 docker-compose 起 PG

#### 1.5 部署改造（2 天）

- Coolify 加 postgres service（用 Coolify 提供的 Postgres 模板）
- 改 docker-compose 加 depends_on
- 改 env 变量 `DATABASE_URL`
- 备份策略：pg_dump cron → 推到 Hetzner Storage Box

### 论文贡献

加入「§7.5 数据库迁移：SQLite → PostgreSQL」章节，~2000 字：
- 迁移动机
- schema 改造点（特别是 SQLite 专属语法）
- 数据迁移流程
- 性能对比（之前 vs 之后单 SKU 回测耗时）
- 部署架构图（含 postgres service）

### 可能的坑

- 🔴 **pgloader 处理 BLOB / 大文本可能慢**：本项目几乎全是 INTEGER / TEXT，问题不大
- 🟡 **PG 时区处理跟 SQLite 不同**：所有 `event_at` 是 TEXT 'YYYY-MM-DD'，迁过去后还是 TEXT，反而最简单（但论文得提一句"为什么不用 PG 的 TIMESTAMP WITH TIMEZONE"）
- 🟡 **alembic 在两种 dialect 下都跑过**：需要 `op.get_bind().dialect.name == 'sqlite'` 守护 SQLite 特定语法
- 🟢 **回测 long-running 任务**：PG 跑 6000+ SKU 的 walk-forward 会比 SQLite 慢一些（PG 连接 overhead 大），但可接受

### 退场标准

- ✅ 708 测试全过
- ✅ alembic upgrade head 在 PG 上跑通
- ✅ 真 DB sanity：跑一遍 4 个 SKU `base_demand_view`，结果跟 SQLite 完全一致
- ✅ 全量 v3 backtest 跑通（PG 版本），分数 vs v2 不超过 5% 漂移
- ✅ Coolify 部署 PG + 应用容器都健康
- ✅ 论文 §7.5 章节落档

## Phase 2 (2026-07): 算法扩展 — Holt-Winters + TSB

### 故事背景

v1/v2 已对 4 个 baseline 做完对比，结论是 NaiveSeasonal52W 最优。但 4 个 baseline 都是「最朴素」的方法。论文目前的批评点：

> "为何不试 Holt-Winters？这是经典的季节性方法，正好补 NaiveSeasonal52W 的不足。"

> "CrostonSBA 是 1972 年的方法，2011 年 Teunter-Syntetos-Babai (TSB) 已证明在 dying SKU 上表现更好。你既然发现了 dying 问题，为什么不直接换 TSB 试试？"

两个评委的疑问都有道理。Phase 2 把这两个补上，让算法对比更完整。

### 主要工作

#### 2.1 Holt-Winters（2 周）

- 引入 `statsmodels` 依赖
- 实现 `HoltWintersModel(season_periods=52)`，需要 ≥ 104 周（2 年）历史
- 对 ≥ 104 周的 SKU（项目里只有 402 个，1.3%）才启用
- 短历史 SKU 退化为 NaiveSeasonal52W
- 加入 walk_forward + 评分

#### 2.2 TSB（1 周）

Teunter-Syntetos-Babai 算法 [3]:

$$
\hat{p}_t = \alpha \cdot \mathbb{1}\{y_t > 0\} + (1 - \alpha) \hat{p}_{t-1}
$$

$$
\hat{z}_t = \alpha y_t + (1 - \alpha) \hat{z}_{t-1} \quad \text{(仅在非零时更新)}
$$

$$
\hat{y}_t = \hat{p}_t \cdot \hat{z}_t
$$

关键差异：TSB 的「需求概率」$\hat{p}$ **每周都更新**（不像 Croston 只在非零事件更新），所以对最后非零事件之后的零周敏感 — **理论上能解决 dying 问题，对照实验值得做**。

#### 2.3 对比实验（1 周）

- v4 backtest：6 个模型 × 1 view = 6 个 run
- 论文 §6.7 新增「HW vs Naive / TSB vs CrostonSBA」两组对比表
- 关键问：TSB 在 dying SKU 上能否不需要分流就工作？（用 v1 的过滤前数据测）

### 论文贡献

- §4.5 加 Holt-Winters 与 TSB 原理
- §6.4 baseline 表扩到 6 列
- §6.5.3 加「TSB 对 dying 的算法层修复 vs 本文数据层修复」对比讨论
- 答辩 PPT 加「6 模型 + 数据/算法层修复对比」幻灯

### 可能的坑

- 🔴 **statsmodels HW 在零值多的序列上不稳**：可能频繁报警告 / NaN，需要预处理（log+1 变换、季节剥离等）
- 🟡 **TSB 跟 SBA 共用 alpha=0.1 可能不公平**：需要 alpha 调优 spike
- 🟡 **statsmodels 单 SKU 跑 walk_forward 慢**：HW fit() 比 NumPy baseline 慢 10x+，6,098 SKU × 几百窗口可能 10h+

### 退场标准

- HoltWinters 6 个 unittest 过（边界 / 短历史 / NaN handling）
- TSB 6 个 unittest 过
- v4 全量 backtest 跑通
- 至少能讲出 1 个「TSB > SBA」的具体 SKU 案例

## Phase 3 (2026-08): ML 集成 — Prophet（可选 LSTM）

### 故事背景

经典统计模型搞完后，评委会问：「ML 方法对比呢？」本科答辩通常不要求 ML，但**有 vs 没有**的论文档次差别明显。

Prophet 是首选：
- Facebook 出品，文档完善
- 适合零售业季节性
- 训练快、可解释（trend + seasonality + holidays）
- 不需要 GPU
- 集成 cost 低（1-2 天）

LSTM 是可选：
- 深度学习招牌
- 但训练慢、参数多、要 GPU
- 对小数据（156 周）容易过拟合
- **本科论文里出现「我们 fine-tuned LSTM」会被怀疑数据量是否合理**

### 主要工作

#### 3.1 Prophet 集成（1 周）

- `pip install prophet`
- 新建 `prophet_model.py`，实现 ForecastModel Protocol
- 关键点：Prophet 需要 DataFrame 输入 `ds` (date) + `y` (value)，要从 `dict[date, int]` 转
- 多步预测：Prophet 天然支持（make_future_dataframe），比 baseline 优势明显
- 节假日：可以加希腊节假日（已有 `public_holidays` 表）！这是 Prophet 的强项

#### 3.2 全量回测对比（3 天）

- v5 backtest: 7 个模型 × 1 view（加 Prophet）
- 关键问：Prophet 跟 NaiveSeasonal52W 谁更准？
- Prophet 通常在长历史 + 强季节 SKU 上明显胜出，短历史/间歇序列上可能不如 Croston

#### 3.3 节假日效应分析（2 天）

- 用 `public_holidays` 表把希腊节假日作为 regressor 加入 Prophet
- 跑 with-holiday vs without-holiday 两版
- 论文加「希腊节假日对预测精度的影响」章节

#### 3.4 LSTM（可选，3 周，仅作者 A 看时间允许）

- 用 PyTorch 简单 LSTM（不上 N-BEATS / Transformer 这种重的）
- 训练数据少（150 周 × 6000 SKU = 90 万样本）够用
- 实际 LSTM 在间歇序列上很难打过 Prophet + Croston
- **如果时间不够就跳过**，论文 §8.4 后续工作里列出来即可

### 论文贡献

- §4.6 加 Prophet 原理 + 实现细节
- §6.4 baseline 表扩到 7-8 列
- §6.8 新增「希腊节假日效应分析」章节
- 答辩 PPT 加「ML vs 经典 baseline」对比

### 可能的坑

- 🔴 **Prophet 安装在 Windows 上麻烦**（依赖 cmdstan），改用 Docker 跑或换 Linux
- 🟡 **prophet 单 SKU fit 比 baseline 慢 5-10x**：6,098 SKU 全跑可能要 2-3h
- 🟡 **LSTM 真做的话调参可能搞死人**：如果时间紧就 defer

### 退场标准

- Prophet 集成 + 6 个 unittest
- v5 全量 backtest 跑通
- 节假日 effect 分析有具体数字（如「春节后 1 周预测精度比无 holiday 模型高 X%」）
