## 1. 项目整体概况

**label-sync**（双端处理）是一个本地 Web 工具，最初为协助希腊小型零售业的标签处理工作而开发，后扩展为完整的进销存数据分析与预测系统。

**部署**: Hetzner VPS + Coolify (Docker) + 内网访问
**数据规模**: 1,361,065 条 inventory_events（2023-01 → 2026-05），27,340 个活跃 SKU
**测试覆盖**: 708 个单测全过（截至 2026-05-15）

## 4. 核心设计决策（答辩可讲）

### 4.1 SKU 销售形态四分类（业务驱动）

经数据 spike 摸底（`_scratch/spike_sku_type_thresholds.py`），把所有 SKU 分四类：

| 类型 | 判定 | 占比（实测） | 处理 |
|---|---|---|---|
| `retail_dominant` | 零售订单（净量 ≤ 24）占比 ≥ 80% | 85.8% | 仅剔单笔异常大单后进时序预测 |
| `mixed` | 零售占比 5%-80% | 1.0% | 同上 + 客户类型过滤（保留 foreign+chinese） |
| `wholesale_only` | 零售订单 < 5 笔 或 占比 < 5% | 13.2% | 不进时序预测（走 wholesale_demand_view 经验分位数路径） |
| `dying` ⭐ | 最后销售距 as_of ≥ 13 周 | 46.3%（覆盖上面三类） | **不进时序预测**（修复 CrostonSBA 系统性高估的关键） |

判定优先级：`unclassified > dying > wholesale_only > retail/mixed`。

### 4.2 退货归并机制（数据正确性）

**问题**: 原 ETL 把退货事件存为负数 qty，直接 `SUM(qty)` 会让退货抵消进周需求，破坏时序结构。

**解决**: 在 `weekly_demand_series` 里按 `document_no` 分组求净量：
- 净量 > 0 → 挂到 doc 内最早事件那一周
- 净量 ≤ 0（孤儿退货 / 完全冲销） → 丢弃，**不落负**
- `document_no` 为空 → 按事件主键独立成 doc

测试: `tests/test_forecast_data.py` 12 个 case 覆盖。

### 4.3 4 个 Baseline 选型理由

| baseline | 思路 | 适用场景 |
|---|---|---|
| **NaiveMean4W** | 最近 4 周均值 | 稳态需求；强基线 |
| **NaiveSeasonal52W** | 去年同周值 | 季节性 SKU |
| **LinearTrend12W** | 最近 12 周线性外推 | 趋势性 SKU |
| **CrostonSBA** | Syntetos-Boylan Approximation（α=0.1） | 间歇需求（零销周率高） |

加 CrostonSBA 的理由：spike 发现空周率 mean 70% 极高（间歇特征显著），传统 Naive 三件套对零值多的序列不友好。

### 4.4 Walk-Forward 滚动回测

- `window_train=13`, `window_test=4`（plan 默认）
- 每次窗口前移 1 周
- 收集 `(step_idx, horizon, predicted, p50, p98, actual)` 五元组
- 评分函数: **MAPE**（mean absolute percentage error，剔零） / **MASE**（mean absolute scaled error，对零稳健） / **Bias**（带符号偏差） / **coverage_p98**（actual ≤ p98 命中率）

加 MASE 的理由: MAPE 在 actual=0 时除以 0 爆炸，间歇序列上不可用；MASE 用 lag-1 naive MAE 做分母，> 1 即比 naive 差。

### 4.5 dying SKU 检测（论文亮点）

**发现过程**:
1. v1 全量回测后，CrostonSBA Bias_avg = +0.89 ❌（其他 baseline 都 < +0.1）
2. spike `_scratch/spike_croston_bias.py` 排查 Top 3 over-forecast SKU
3. 发现 3 个 SKU 都是 **stop-sale SKU**：最后销售在 13-63 周前
4. 原因: CrostonSBA EWMA size 用 α=0.1 衰减太慢，「记忆」历史大单 → 在已停售 SKU 上仍预测高值
5. **修复**: 加 `dying` SKU 状态（最后销售距 as_of ≥ 13 周），dying SKU 不进时序预测，直接 series=None

**为什么 N=13**: spike `_scratch/spike_dying_threshold.py` 摸底，发现 46.3% SKU 会被标 dying；阈值跟 plan §3.5「新品规则 days_since_first_sale < 90」对称（90 天 ≈ 13 周）。

## 5. 回测实测结果（v1，2026-05-14 完成）

**测试集**: 27,340 active SKUs，全部 4 baseline × view='base_demand'，每次跑 ~118 min，总 7.65h。

| 模型 | n_scored | MAPE_med | MASE_med | Bias_avg | cov@p98 |
|---|---|---|---|---|---|
| NaiveMean4W | 6,098 | 0.96 | **0.941** ✅ | +0.04 | 89.7% |
| **NaiveSeasonal52W** | 6,098 | **0.85** ✅ | 0.948 | +0.08 | **92.1%** ✅ |
| LinearTrend12W | 6,098 | 1.05 | 1.056 ❌ | +0.01 | 91.3% |
| CrostonSBA | 6,098 | 0.93 | **1.141** ❌ | **+0.89** ❌ | 92.6% |

**结论**:
- NaiveSeasonal52W 综合最优 baseline（MAPE 最低 + coverage 最高 + MASE < 1）
- LinearTrend12W 全面最差（趋势外推在间歇序列上 amplify 噪音）
- CrostonSBA 反常的 +0.89 高估 → 引出 dying SKU 发现 → 修复后预期回落（v2 实测验证）

v2 实测数据待补（重跑中，ETA 5-6h）。

## 6. 数据库 Schema

新增 2 表（alembic `b9e1c4f8a3d2_add_backtest_tables`）:

```sql
CREATE TABLE backtest_runs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    created_at TEXT DEFAULT (datetime('now', 'localtime')),
    model_name TEXT NOT NULL,
    view TEXT NOT NULL,           -- 'all' | 'base_demand'
    window_train INTEGER NOT NULL,
    window_test INTEGER NOT NULL,
    min_weeks INTEGER NOT NULL,
    n_skus_total INTEGER DEFAULT 0,
    n_skus_scored INTEGER DEFAULT 0,
    notes TEXT
);

CREATE TABLE backtest_results (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id INTEGER NOT NULL REFERENCES backtest_runs(id),
    product_barcode TEXT NOT NULL,
    sku_type TEXT NOT NULL,
    n_weeks_train INTEGER NOT NULL,
    n_weeks_test INTEGER NOT NULL,
    mape REAL, mase REAL,
    bias REAL NOT NULL,
    coverage_p98 REAL NOT NULL,
    mean_actual REAL NOT NULL,
    mean_predicted REAL NOT NULL
);
CREATE INDEX idx_backtest_results_run_id ON backtest_results(run_id);
CREATE INDEX idx_backtest_results_barcode ON backtest_results(product_barcode);
```

## 7. HTTP API（回测）

| 端点 | 方法 | 用途 |
|---|---|---|
| `/analytics/backtest/run` | POST | 启动一次回测（同步），返回 run_id |
| `/analytics/backtest/runs` | GET | 列出最近 50 次 run 元信息 |
| `/analytics/backtest/results?run_id=N` | GET | 单 run 的 per-SKU 分数 |
| `/analytics/backtest/compare?run_a=A&run_b=B` | GET | 两 run 的 per-SKU 差异 + summary（improved/worsened/unchanged + median_mase_delta） |

POST body 示例:
```json
{
  "model_name": "NaiveMean4W",
  "end_date": "2026-05-13",
  "weeks": 156,
  "view": "base_demand",
  "window_train": 13,
  "window_test": 4,
  "min_weeks": 20,
  "notes": "thesis run"
}
```
